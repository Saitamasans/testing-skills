# 增量 Skill：正式服验证用例生成 Skill

## 一、Skill 定位

本 Skill 用于已经上线到正式服、预发布环境、生产环境、灰度环境后的功能验证。

它不替代测试服全量测试用例，也不用于覆盖所有边界、异常、并发、性能和破坏性场景。

它的目标是生成一套：

1. 能证明核心功能已在正式服可用的验证用例
2. 能覆盖主流程闭环的最小充分执行集
3. 尽量少写库、少造数、少产生脏数据
4. 不影响真实用户、不改变线上关键配置
5. 不执行高风险破坏性测试
6. 可回溯、可归档、可向产品/研发/运营证明上线状态的正式服验证资产

---

## 二、触发场景

当用户出现以下表达时，触发本 Skill：

| 触发语         | 说明               |
| ----------- | ---------------- |
| 正式服验证用例     | 为已上线功能设计生产环境验证用例 |
| 线上验证用例      | 同正式服验证           |
| 生产环境冒烟      | 生成最小正式服冒烟验证集     |
| 上线后验证       | 功能已发布后做线上检查      |
| 灰度验证        | 灰度环境或部分用户开放后的验证  |
| 线上回归        | 已上线功能的轻量回归       |
| 正式服不能测太多    | 需要控制测试范围和脏数据     |
| 已经上正式服了，怎么测 | 进入正式服验证策略        |

---

## 三、核心原则

正式服验证不是测试服全量回归。

必须遵循以下原则：

| 原则      | 说明                            |
| ------- | ----------------------------- |
| 主流程优先   | 优先验证入口、核心操作、核心结果、记录/后台/日志闭环   |
| 少量写操作   | 能只读就只读；必须写库时，只用最小可控数据         |
| 低价值数据   | 奖励、支付、礼物、积分、抽奖等尽量使用低价值、可接受的数据 |
| 可识别数据   | 测试账号、测试房间、测试昵称、测试备注必须可识别      |
| 可回滚/可归档 | 产生的数据要可查询、可解释、可清理或可归档         |
| 不影响用户   | 不改全局配置、不影响真实用户路径、不制造异常流量      |
| 不做破坏性测试 | 并发、刷接口、风控、批量注册、恶意越权等默认不在正式服执行 |
| 分层验证    | 页面、接口、数据、记录、监控按风险选择，不机械全测     |
| 风险留痕    | 不适合正式服执行的用例要单独列出，不要直接删除       |
| 验证可诊断   | 每条用例要能判断失败原因，不写“检查是否正常”这类空泛描述 |

## 四、正式服验证输入判断

拿到需求或测试服用例后，先判断正式服验证类型：

| 类型          | 特征                   | 输出策略                 |
| ----------- | -------------------- | -------------------- |
| P0 主链路正式服验证 | 核心功能必须在线上跑通          | 必须生成正式服验证用例          |
| P1 低风险补充验证  | 不一定阻断上线，但能提高信心       | 选择性生成                |
| P2 展示/兼容验证  | 低风险、只读、无写库           | 可轻量纳入                |
| 禁止正式服执行     | 会污染数据、影响用户、改变配置、触发风控 | 不生成执行用例，放入“不建议正式服执行” |
| 条件执行        | 需要运营/研发/客服/后台支持      | 标注执行条件和风险            |

## 五、正式服验证分层模型

### 1. L0 只读验证

不产生业务数据，仅查看页面、入口、文案、配置、记录。

| 场景          | 是否推荐正式服执行 |
| ----------- | --------- |
| 活动入口是否展示    | 推荐        |
| Banner 是否跳转 | 推荐        |
| 活动时间、规则说明展示 | 推荐        |
| 用户记录查询      | 推荐        |
| 后台记录查询      | 推荐        |
| 配置只读检查      | 推荐        |

### 2. L1 低风险写操作

会产生少量数据，但可控、可识别、影响小。

| 场景           | 是否推荐正式服执行   |
| ------------ | ----------- |
| 内部测试账号绑定邀请码  | 推荐，但只执行 1 次 |
| 测试房间进入/发言/上麦 | 推荐，但控制次数    |
| 低价值礼物直刷      | 条件执行        |
| 抽奖 1 次       | 条件执行        |
| 生成一条可识别记录    | 推荐          |

### 3. L2 受控高风险写操作

可能涉及奖励、资产、库存、后台配置，需要审批或灰度。

| 场景       | 是否推荐正式服执行      |
| -------- | -------------- |
| 抽奖发奖     | 只允许最小次数        |
| 修改后台活动配置 | 默认不建议，除非灰度配置   |
| 客服补偿     | 仅在真实失败或演练场景下执行 |
| 贡献值大额触发  | 不建议正式服构造       |

### 4. L3 禁止/不建议正式服执行

| 场景          | 原因               |
| ----------- | ---------------- |
| 批量注册新用户     | 污染用户数据           |
| 高频抽奖/批量抽奖   | 污染奖池和奖励记录        |
| 并发压测        | 影响真实用户           |
| 恶意刷风控       | 可能触发真实处罚         |
| 修改活动开始/结束时间 | 影响全量用户           |
| 关闭正式服活动开关   | 影响真实用户           |
| 构造百万贡献值     | 资损和脏数据风险高        |
| 非授权越权攻击     | 安全风险，应在测试/预发环境执行 |
| 大量接口重放      | 可能触发限流、风控、误告警    |
| 删除/回滚线上数据   | 高风险，需专项审批        |

## 六、正式服验证用例设计流程

生成正式服验证用例时，必须按以下步骤处理：

### 第一步：确认正式服验证目标

| 问题                  | 说明                   |
| ------------------- | -------------------- |
| 验证什么功能已经上线？         | 明确活动、页面、接口、后台、任务、奖励等 |
| 验证范围是全量、灰度还是指定账号？   | 决定账号和入口              |
| 是否允许写库？             | 决定是否做绑定、发奖、抽奖        |
| 是否允许真实发奖/扣资产？       | 决定是否执行奖励闭环           |
| 是否有内部测试账号？          | 降低污染                 |
| 是否有测试房间/测试礼物/低价值道具？ | 控制风险                 |
| 是否需要后台/日志/监控配合？     | 保证可诊断                |

### 第二步：从测试服全量用例中筛选正式服用例

优先保留：

1. 入口可见性
2. 页面可访问性
3. 核心主流程
4. 最小数据写入闭环
5. 奖励或状态结果验证
6. 记录查询
7. 后台/日志/监控核对
8. 关键失败态的轻量验证

默认剔除：

1. 大量边界组合
2. 批量异常输入
3. 并发压测
4. 高价值奖励验证
5. 后台全局配置修改
6. 恶意越权攻击
7. 风控刷量
8. 破坏性数据构造

### 第三步：确定正式服测试数据策略

| 数据类型 | 推荐策略                     |
| ---- | ------------------------ |
| 测试账号 | 使用内部账号，昵称或备注包含“正式服验证”    |
| 邀请关系 | 仅准备 1 组邀请人 + 被邀请人        |
| 房间   | 使用内部测试房间或低流量房间           |
| 礼物   | 使用低价值礼物，避免高价值资产          |
| 抽奖   | 只抽 1 次，验证扣点、发奖、记录闭环即可    |
| 奖励   | 优先验证低价值奖励或后台记录           |
| 后台配置 | 只读核对，不改正式配置              |
| 日志   | 使用测试账号 ID、traceId、时间范围查询 |
| 数据清理 | 能清理则清理，不能清理则归档说明         |

### 第四步：生成正式服验证用例

正式服用例必须满足：

1. 每条用例步骤少
2. 每条用例目标明确
3. 不重复验证低价值点
4. 写库动作最小化
5. 每个写库动作都有记录核对
6. 每个风险动作都有备注说明
7. 每个失败态都尽量采用不产生副作用的方式验证
8. 对不能在线上执行的测试点，放入“不建议正式服执行”清单

## 七、正式服验证用例表字段

正式服测试用例表统一使用以下字段：

| 用例 ID | 所属模块 | 用例标题 | 验证功能点 | 前置条件 | 测试步骤 | 预期结果 | 优先级 | 执行结果（通过 / 不通过 / 未执行） | 备注 |
| ----- | ---- | ---- | ----- | ---- | ---- | ---- | --- | -------------------- | -- |

字段要求：

1. 用例 ID 使用模块缩写-三位序号，例如 PROD-001、BIND-001、DRAW-001。
2. 每个模块前插入【模块分割行】。
3. “验证功能点”必须用 1、2、3 编号列出该用例实际验证什么。
4. “前置条件”“测试步骤”“预期结果”必须编号。
5. “执行结果（通过 / 不通过 / 未执行）”默认填写“未执行”。
6. 如果生成 Excel / Word / WPS，字体设置为黑体。
7. 不新增“测试数据”“是否冒烟”“是否回归”“是否自动化”等列。
8. 测试数据要写入前置条件、测试步骤或备注中。

## 八、正式服验证模块建议

根据功能复杂度选择模块，不要求机械全用。

| 模块           | 适用场景                 |
| ------------ | -------------------- |
| 正式服入口与展示验证   | 页面入口、Banner、活动页、规则文案 |
| 正式服账号与权限验证   | 登录态、测试账号、访问权限、灰度资格   |
| 正式服最小业务闭环    | 绑定、提交、领取、创建、支付等核心动作  |
| 正式服奖励/资产闭环   | 积分、点数、礼物、优惠、余额、抽奖等   |
| 正式服记录与后台核对   | 前台记录、后台记录、日志、报表      |
| 正式服前后端限制轻量验证 | 状态拦截、接口拒绝、无副作用失败态    |
| 正式服监控与告警观察   | 错误日志、接口成功率、异常告警      |
| 正式服收尾与数据归档   | 测试数据标记、清理、归档说明       |

---

## 九、正式服验证优先级规则

| 优先级 | 判断标准                             |
| --- | -------------------------------- |
| P0  | 正式服必须验证的核心入口、核心主流程、核心数据结果、资损相关闭环 |
| P1  | 重要但可条件执行的记录、后台、日志、轻量异常、兼容验证      |
| P2  | 低风险展示、文案、弱网、低频兼容、非阻断验证           |

正式服 P0 不等于测试服 P0。
测试服 P0 里的并发、越权、批量边界、风控刷量，不一定适合正式服执行。

---

## 十、正式服验证中的前后端限制策略

正式服仍要验证前后端一致性，但不能使用高破坏方式。

| 场景           | 正式服策略                    |
| ------------ | ------------------------ |
| 活动未开始        | 可以通过只读状态或灰度账号验证，不强行改正式时间 |
| 活动已结束        | 可以在自然结束后验证，或用灰度/测试活动验证   |
| 点数不足         | 推荐验证，副作用低                |
| 未登录/token 过期 | 推荐验证，副作用低                |
| 非法参数         | 可通过单次接口验证，避免批量 fuzz      |
| 越权           | 不做攻击式扫描，只做受控单例验证         |
| 连点           | 可做 1 次轻量连点，不做高频重放        |
| 并发           | 正式服默认不做，除非灰度环境和研发许可      |
| 库存仅剩 1       | 正式服默认不构造，避免影响真实奖池        |
| 活动开关关闭       | 不改正式开关，除非维护窗口或灰度配置       |

## 十一、不建议正式服执行清单

正式服用例输出时，必须单独列出“不建议正式服执行”内容。

输出格式：

| 不建议执行项 | 原因 | 建议替代验证方式 |
| ------ | -- | -------- |

常见内容：

| 不建议执行项   | 原因            | 替代方式            |
| -------- | ------------- | --------------- |
| 批量注册测试用户 | 污染正式用户数据      | 使用已有内部测试账号      |
| 高频抽奖     | 消耗真实奖池和点数     | 只抽 1 次，其他在测试服验证 |
| 并发抽奖     | 可能影响奖池、日志和告警  | 测试服完成并发验证       |
| 修改正式活动时间 | 影响真实用户        | 只读核对配置          |
| 修改正式活动开关 | 影响活动入口和参与     | 后台只读核对          |
| 刷贡献值     | 资损和业务污染风险高    | 测试服验证，正式服只查统计   |
| 触发风控     | 可能误封内部账号或扰动风控 | 测试服/风控沙箱验证      |
| 大量接口非法参数 | 可能触发安全告警      | 单例验证或测试服验证      |
| 删除测试数据   | 存在误删风险        | 标记归档，必要时走审批清理   |

---

## 十二、正式服验证输出结构

生成正式服验证资产时，建议输出以下内容：

1. 正式服验证策略说明
2. 正式服验证用例表
3. 不建议正式服执行清单
4. 测试数据与账号准备说明
5. 执行前检查清单
6. 执行后收尾清单
7. 风险与观察项

## 十三、执行前检查清单

| 检查项            | 是否必须 |
| -------------- | ---- |
| 确认正式服版本已发布     | 是    |
| 确认活动/功能开关状态    | 是    |
| 确认测试账号可登录      | 是    |
| 确认测试账号不是真实用户   | 是    |
| 确认是否允许写库       | 是    |
| 确认是否允许发奖/扣点/支付 | 是    |
| 确认测试房间/测试对象    | 是    |
| 确认后台或日志查询权限    | 建议   |
| 确认失败时联系人       | 是    |
| 确认执行时间窗口       | 是    |
| 确认是否需要运营/客服知会  | 视情况  |
| 确认监控大盘无异常      | 建议   |

## 十四、执行后收尾清单

| 收尾项      | 说明              |
| -------- | --------------- |
| 记录测试账号   | 便于后续追溯          |
| 记录测试时间   | 便于查日志           |
| 记录产生的数据  | 绑定关系、抽奖记录、奖励记录等 |
| 核对前后台一致性 | 前端、后台、日志、背包/资产  |
| 观察告警     | 接口错误率、异常日志、告警平台 |
| 标记脏数据    | 不能清理的要归档说明      |
| 清理可清理数据  | 必须走审批或后台能力      |
| 输出验证结论   | 通过/不通过/阻塞/有风险上线 |
| 归档截图和日志  | 保留上线验证证据        |

## 十六、正式服验证结果判断

| 结果    | 判断标准                            |
| ----- | ------------------------------- |
| 通过    | P0 用例全部通过，P1 无阻断问题，监控无异常        |
| 有风险通过 | P0 通过，但存在 P1/P2 问题或监控轻微异常       |
| 阻塞    | 任一 P0 主流程失败、资损异常、数据不一致、接口严重报错   |
| 回滚建议  | 主流程不可用、发奖/扣款/资产异常、影响真实用户、告警持续扩大 |
| 待观察   | 功能可用但存在异步延迟、日志异常、少量用户反馈待确认      |

## 十七、正式服验证最终自检

输出正式服验证用例前，必须检查：

1. 是否只保留正式服适合执行的最小充分用例。
2. 是否覆盖入口、主流程、结果、记录/后台、监控闭环。
3. 是否避免批量造数、大量写库、高频接口、并发压测。
4. 是否明确哪些测试不建议正式服执行。
5. 是否每条写库用例都有数据归档或清理说明。
6. 是否每条用例都有“验证功能点”。
7. 是否每条用例前置条件、步骤、预期结果都有编号。
8. 是否执行结果默认是“未执行”。
9. 是否说明测试账号、测试数据、测试时间窗口。
10. 是否能在失败时快速定位是前端、接口、数据、配置、权限还是环境问题。
11. 如果导出为文件，测试用例表是否使用黑体。
