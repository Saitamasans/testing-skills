# 多源测试审计 Skill v0.1

## 定位与适用范围

关联需求、接口文档、客户端、后端与 Admin 等多源材料，完成能力分级、业务链选择、静态审计线索和后续验证计划。

适用于需要先盘点材料、还原跨源业务链、选择一个深入范围并生成阶段 A 静态审计交付物的任务。不用于普通单接口用例生成、已有用例质量评审，或直接执行接口、数据库和浏览器自动化。

本工作流与其他 Skill 互斥路由：用户确认前只盘点和展示候选链，不生成深入计划；每轮最多一个被选链可以进入阶段 A 深入分析。

## 硬规则

- 阶段 A 只读；不得调用业务接口、数据库或浏览器，不得修改业务代码。
- 先登记材料并判断 M1～M4：仅文档为 M1；存在源码为 M2；源码加测试环境与账号为 M3；再有数据库、日志、缓存、消息或其他副作用证据为 M4。
- 从材料中提出恰好 3 条候选业务链，等待用户明确选择；只深入被选链。
- 明确区分材料事实、多源推断、通用测试启发式、用户确认规则和规则未知。未知规则的预期结果必须为“规则待确认”。
- 静态线索不得称为已确认 Bug、漏洞或资损。

## 输入与固定流程

可输入需求或规则、Apifox/Postman/OpenAPI/Swagger、客户端/后端/Admin 源码、日志/HAR/抓包。源码优先使用解压后的只读目录。

材料盘点 → M1～M4 判断 → 项目画像 → 多源关联 → 3 条候选业务链 → 用户选择 → 只深入选中链 → 静态线索与阶段 B 审批计划 → 聊天摘要与固定四表 Excel。

阶段 A 输出聊天重点、完整机器 JSON 和固定四表 Excel：审计总览、多源关联、审计计划、执行结果。审计计划固定 10 列：计划编号、审计场景、对应接口/业务动作、规则或材料依据、前置条件与账号角色、变异参数、黑盒验证步骤、预期结果、核验点/副作用证据、风险/授权范围与审批状态。执行结果全部为“未执行”，原因是尚未获得阶段 B 审批。

不生成正式测试用例；阶段 A 只提供带来源依据的静态审计计划和后续审批计划。

## 调用示例

调用 `multi-source-test-audit`：请只读分析这份需求、Apifox 导出和三套解压源码，先盘点材料并判断 M 等级，再给出 3 条候选业务链；我选择后只深入该链并生成阶段 A 四表 Excel，不调用接口或数据库。

## 自检

- [ ] 材料、M 等级、3 条候选链均有来源。
- [ ] 已获得用户选择，深入内容没有混入其他候选链。
- [ ] 每条计划有稳定来源；启发式和未知规则未冒充事实。
- [ ] Excel 仅有固定 4 Sheet，审计计划为 10 列，执行结果未执行。
- [ ] 未进行动态操作，未把静态线索定性为确认缺陷。
