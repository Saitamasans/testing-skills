# 八个测试 Skill 全流程演示与双版本录屏设计

日期：2026-07-15

仓库：`Saitamasans/testing-skills`

演示项目：`SkillMart` 电商订单演示系统

## 1. 背景

现有 Todo 自测只证明了第八个 `web-api-test-execution-evidence` Skill 能够对一组预制 Web/API 输入执行基础闭环。该自测没有真实完整调用原七个 Skill，用例输入由脚本生成，也没有把成功截图、完整需求演进和七 Skill 路由组织成适合公开展示的材料。

本设计建立一个从模糊需求到自动执行结束的完整公开演示。演示必须真实调用原七个 Skill，再由第八个 Skill 执行其中五个生成类 Skill 产出的正式测试用例（Test Cases），保留原始材料、执行证据和视频记录。

## 2. 目标

必须实现：

1. 从一份存在真实缺口的 PRD v0 开始，展示需求如何被澄清并形成可测的 PRD v1；
2. 分七次独立调用原七个 Skill，不把互斥 Skill 塞进同一次调用；
3. 五个生成类 Skill 分别产出正式测试用例（Test Cases）及同源 Excel、HTML；
4. 第八个 Skill 自动执行五套正式测试用例（Test Cases）；
5. 真实展示通过、不通过、待定和未执行四种业务状态；
6. 为成功、失败、待定、API 请求响应、数据清理和报告一致性保留可核验材料；
7. 交付完整未剪辑静音录屏和 20 分钟以内的静音字幕精剪版；
8. 让公开用户能根据演示材料、提示词和仓库内容复现完整流程。

## 3. 不在本次范围内

- 不合并、拆分、删除或重命名原七个 Skill；
- 不改变用户原有中文表达习惯和十列字段顺序；
- 不把七个 Skill 伪装成一次并行调用；
- 不接入真实支付、真实库存、真实用户或真实正式服；
- 不录制麦克风、系统声音或 AI 旁白；
- 不把视频作为测试证据的唯一来源；
- 不为了控制视频时长缩减 Skill 能力、测试用例（Test Cases）内容或执行准确性；
- 不在本次设计阶段实现网页、生成正式文件或开始录制。

## 4. 双语展示规则

公开说明、演示导航、视频字幕和讲解文档提到测试资产时使用：

- 单条：测试用例（Test Case）；
- 一批或整体：测试用例（Test Cases）。

该规则用于显示名称，不修改以下稳定契约：

- Skill 名称；
- 现有文件路径和触发口令；
- 正式测试用例（Test Cases）的十列中文字段；
- 每条测试用例（Test Case）的中文业务标题。

## 5. 方案选择

采用“单一业务主线逐步进化”。

未采用：

- 七个互不相关的小案例：路由清晰，但视频叙事碎片化；
- 同一输入横向喂给所有 Skill：容易制造功能重复印象，并违反互斥路由设计；
- 只展示最终报告：无法证明七个 Skill 的实际作用和输入演进。

## 6. 演示业务

### 6.1 产品名称

`SkillMart` 电商订单演示系统。

### 6.2 业务能力

- 商品列表与库存展示；
- 优惠券资格校验；
- 创建订单并锁定库存；
- 查询自己的订单；
- 支付回调；
- 取消待支付订单并释放库存；
- 页面、接口、订单状态和库存结果一致性核对。

### 6.3 PRD v0

PRD v0 有明确业务目标，但故意缺少足以阻塞正式测试用例（Test Cases）生成的核心规则：

- 操作角色和数据归属；
- 优惠券门槛和有效期判定时点；
- 库存锁定、释放时机；
- 创建订单幂等规则；
- 支付回调重复处理规则；
- 订单状态及合法迁移；
- 页面、接口和数据验收口径。

需求澄清章节必须真实输出 `P0 ⛔ BLOCKING`，不得提前生成正式测试用例（Test Cases）。

### 6.4 PRD v1 已确认规则

产品确认后形成 PRD v1：

1. 普通用户只能查询、支付和取消自己的订单；
2. 商品库存大于零才能下单，创建订单时锁定对应数量；
3. 优惠券规则为满 100 减 20，每个订单最多使用一张；
4. 创建订单必须携带幂等键，相同用户使用相同幂等键重复请求时只能产生一个订单并锁定一次库存；
5. 订单状态为 `PENDING_PAYMENT`、`PAID`、`CANCELLED`；
6. 待支付订单可以取消并释放库存，已支付和已取消订单不能再次取消；
7. 支付回调必须幂等，重复回调不能产生重复支付结果或重复业务副作用；
8. 用户页面、订单查询接口、订单记录和库存结果必须一致；
9. 所有错误码、状态码、鉴权和清理口径在接口契约中明确。

### 6.5 保留的真实缺陷

`POST /api/orders` 故意错误处理幂等键。相同用户使用相同幂等键提交相同订单时，系统会生成两张订单并重复锁定库存。

对应测试用例（Test Case）必须真实执行并判定为“不通过”。多个测试用例（Test Cases）命中该问题时，最终缺陷摘要按根因聚合为一个缺陷，保留各自证据但不重复计算 Bug 数量。

### 6.6 保留的需求歧义

优惠券恰好在提交边界过期时存在两份冲突依据：

- 产品确认记录要求按用户点击提交的客户端时间判断；
- 接口契约要求按服务端收到请求的时间判断。

该边界不影响其他核心流程进入正式测试设计，但对应测试用例（Test Case）必须记录两份证据。第八个 Skill 真实执行后，无论实现选择哪种口径，最终业务状态均为“待定”，不得标记为研发 Bug。

## 7. 演示系统架构

### 7.1 技术边界

- 使用本地单进程服务；
- 页面使用原生 HTML、CSS 和 JavaScript；
- API 使用 Node.js 内置 HTTP 能力；
- 业务数据保存在内存中；
- 不依赖真实数据库、支付服务、账号密码或外部密钥；
- 服务只监听 `127.0.0.1`；
- 每套测试用例（Test Cases）执行前使用本地专用重置入口恢复种子数据。

### 7.2 页面

- 商品与库存列表；
- 优惠券输入与校验结果；
- 创建订单操作区；
- 订单详情和状态；
- 支付回调模拟入口；
- 取消订单入口；
- 明确的成功、失败和当前状态反馈。

### 7.3 API

| 接口 | 作用 |
|---|---|
| `GET /api/products` | 查询商品和库存 |
| `GET /api/coupons/{code}/eligibility` | 校验优惠券资格和有效期 |
| `POST /api/orders` | 创建订单并锁定库存，包含故意保留的幂等缺陷 |
| `GET /api/orders/{id}` | 查询指定用户自己的订单 |
| `POST /api/payments/callback` | 模拟幂等支付回调 |
| `POST /api/orders/{id}/cancel` | 取消待支付订单并释放库存 |
| `GET /api/health` | 只读健康检查 |
| `GET /api/config/public` | 只读公开配置核对 |
| `POST /__test/reset` | 仅限本地执行的数据重置入口 |

### 7.4 主数据流

```text
商品查询
→ 优惠券校验
→ 创建订单并锁定库存
→ 支付回调
→ 查询订单状态
→ 核对订单与库存
→ 重置本地测试数据
```

## 8. 七个 Skill 的独立调用

每章必须保存原始输入、实际提示词、Skill 路由说明、完整回答和关键截图。每次只设置一个主 Skill，不加载辅助 Skill。

| 章节 | 主 Skill | 输入 | 产出 |
|---|---|---|---|
| 1 | `requirement-clarification-test` | PRD v0 | 开测准入、`P0 ⛔ BLOCKING`、已确认内容、产品核对问题、本轮不纳入；不生成测试用例（Test Cases） |
| 2 | `requirement-test-workbench` | PRD v1 和产品确认记录 | 完整测试设计、正式十列测试用例（Test Cases）、报告 JSON、Excel、HTML |
| 3 | `single-api-test-full` | 完整 `POST /api/orders` 契约 | 完整门禁、八维分析、幂等/并发/权限/副作用测试用例（Test Cases）、Excel、HTML |
| 4 | `single-api-test-concise` | 完整 `GET /api/orders/{id}` 契约和显式“精炼版”口令 | 低上下文但可执行的精炼测试用例（Test Cases）、Excel、HTML |
| 5 | `multi-api-flow-test` | 商品、优惠、下单、回调、查询、取消的真实顺序和字段依赖 | 链路模型、可观测性、清理策略、联合测试用例（Test Cases）、Excel、HTML |
| 6 | `production-verification-test` | 明确声明为线上验证目标，但不提供生产写入四项门禁 | 写操作范围阻塞、L0 只读正式测试用例（Test Cases）、不建议线上执行清单、Excel、HTML |
| 7 | `test-case-quality-audit` | PRD v1、产品确认、接口契约和需求工作台生成的主测试用例（Test Cases） | 完整审计、问题 ID、证据、漏测/冗余判断、门禁结论；在提示词明确授权后按问题 ID 生成修订映射 |

正式服章节不猜测环境性质。提示词明确声明验证目标，实际运行地址仍是本地隔离系统，不连接真实生产系统。

## 9. 第八个 Skill 执行闭环

### 9.1 执行对象

第八个 `web-api-test-execution-evidence` Skill 分别执行以下五套正式测试用例（Test Cases）：

1. 需求工作台；
2. 单接口完整版；
3. 单接口精炼版；
4. 多接口链路；
5. 正式服验证的 L0 只读集合。

每套单独编译 Manifest、审批、运行和校验，执行前重置本地种子数据，禁止不同套件互相污染。

### 9.2 必须先修复的第八个 Skill 缺口

1. 首次准备 Runner 不得假定 Node.js 可执行文件旁边存在 npm CLI；必须兼容 Codex 内置 Node；
2. 成功 Web 动作和成功测试用例（Test Cases）必须导出独立 PNG，并在 `run-result.json` 登记证据；
3. API 动作必须保存脱敏后的请求、响应、状态码、断言和清理证据；
4. 期望口径冲突的测试用例（Test Case）必须在动作执行和证据收集后判为“待定”；
5. 运行开始时间必须在执行前记录，完成时间在执行结束后记录；
6. 多个失败测试用例（Test Cases）允许按确定的根因键聚合缺陷摘要，但不得丢失单条证据。

### 9.3 业务状态

| 状态 | 演示含义 |
|---|---|
| 通过 | 实际结果满足已确认需求和接口契约 |
| 不通过 | 幂等缺陷导致重复订单和重复库存锁定 |
| 待定 | 已执行并取得实际结果，但产品和接口口径冲突 |
| 未执行 | 明确需要人工视觉判断或外部监控权限，Runner 不伪造执行结论 |

### 9.4 运行状态与业务状态分离

- 服务、浏览器、网络或 Runner 启动失败属于基础设施错误；
- 定位器失败属于执行器或页面契约问题，不自动算业务失败；
- 缺账号、权限或监控条件时使用 blocked 或 manual required；
- 清理失败必须提升为交付风险；
- 报告一致性失败时禁止宣称完成。

## 10. 证据设计

### 10.1 Web 证据

- 页面初始状态；
- 关键输入完成状态；
- 提交前后状态；
- 成功断言画面；
- 不通过实际结果；
- 待定实际结果；
- 失败定位信息；
- 完整 Playwright Trace。

### 10.2 API 证据

API 没有网页画面，不伪造浏览器截图。每个动作保存：

- 请求方法和路径；
- 脱敏请求头和请求体；
- 响应状态码和脱敏响应体；
- 字段提取结果；
- 断言期望与实际值；
- 清理请求和清理结果；
- 可视化 API 流水页，内容可回指原始事件日志。

### 10.3 可追溯信息

- 输入版本；
- Skill 名称和调用理由；
- 原始提示词；
- Manifest Hash；
- Run ID；
- 逐动作时间线；
- 每个证据文件的路径和 SHA-256；
- Excel、HTML、JSON 一致性结果。

## 12. 视频设计

### 12.1 录制环境

- 使用专门的干净 Windows 录制桌面；
- 固定分辨率 `1920×1080`，30 FPS；
- 只展示 Codex、演示网页、正式测试用例（Test Cases）和执行报告；
- 开启勿扰，关闭通知预览；
- 使用演示专用路径和演示数据；
- 录制前扫描 Token、密码、个人聊天、私人文件名和无关窗口；
- 全程静音，不采集系统音频和麦克风。

### 12.2 完整未剪辑版

`完整未剪辑录屏_Raw-Full-Session.mp4` 从 PRD v0 开始持续录制到最终报告验收结束。保留真实等待、失败、修复、重试和下载过程，不强行限制时长，不做内容剪切。仅允许在开始录制前完成隐私准备，不允许通过后期删除不利画面来冒充一次成功。

### 12.3 20 分钟精剪版

精剪版硬上限 20 分钟，目标约 19 分钟。只压缩画面停留、等待、下载和重复浏览，不压缩实际 Skill 能力或正式测试用例（Test Cases）产物。

| 时间 | 内容 |
|---|---|
| 00:00-00:40 | 项目、安装与八 Skill 链路 |
| 00:40-02:20 | PRD v0 和需求澄清门禁 |
| 02:20-04:20 | PRD v1 和需求工作台 |
| 04:20-06:20 | 单接口完整版与精炼版对比 |
| 06:20-08:20 | 多接口链路和字段传递 |
| 08:20-09:40 | 正式服写操作门禁 |
| 09:40-11:30 | 测试用例审计（Test Case Audit）与修订 |
| 11:30-16:20 | 第八个 Skill 可见自动执行 |
| 16:20-18:20 | 四状态、证据和报告验收 |
| 18:20-19:00 | 安装口令与复现总结 |

### 12.4 字幕和剪辑材料

- 精剪版使用中文字幕；
- 提到测试资产时遵循测试用例（Test Case）/测试用例（Test Cases）双语规则；
- 不生成 AI 旁白；
- 提供 SRT 字幕、章节时间点、原始片段边界和建议剪辑清单；
- 用户可以基于未剪辑版重新剪辑或自行配音。

## 13. 隐私与安全

- 不录制当前包含历史聊天的任务；录制时使用专门的公开演示任务；
- 不显示个人账号、邮箱、微信、浏览器个人标签页或通知；
- 不将 Token、密码、连接串写入提示词、Manifest、日志、截图、字幕或视频；
- 演示账号全部为本地虚拟身份；
- 正式服章节只展示安全门禁和本地 L0 只读执行；
- 原始录屏交付前只做隐私扫描，不做内容剪辑；发现敏感信息时整段作废并重新录制，不能通过打码后的版本冒充原始未剪辑版。

## 14. 错误处理

| 错误 | 处理 |
|---|---|
| Runner 首次准备失败 | 保留原始错误，不继续假装执行；修复后重新开始可追溯运行 |
| 浏览器不可见或无法启动 | 标记基础设施阻塞，不切换为无界面后宣称可见执行 |
| 定位器失效 | 生成修复建议，确认后再应用，不静默自愈 |
| API 请求失败 | 保存请求/响应和错误类型，区分业务失败与网络/执行器错误 |
| 待定口径未携带证据 | 禁止判为待定，退回输入准备 |
| 数据清理失败 | 报告突出显示并停止下一套执行，先恢复种子状态 |
| Excel/HTML/JSON 不一致 | 报告门禁失败，禁止交付 |
| 录制工具中断 | 保留损坏诊断，重新从头录制未剪辑版；不能拼接后命名为原始连续录屏 |
| 视频出现敏感信息 | 作废原始录屏并重新录制 |

## 15. 测试策略

### 15.1 演示系统

- 商品、优惠券、订单、库存和状态迁移单元测试；
- 正常创建、取消、支付回调和权限隔离 API 测试；
- 对故意保留的幂等缺陷设置明确的已知失败基线，避免其他实现错误混入；
- 重置入口只能从本地访问，并能恢复确定性种子状态；
- 页面桌面和窄屏视口截图检查，无重叠、断字和空白画面。

### 15.2 Skill 路由与产物

- 七个提示词分别只命中预期主 Skill；
- 需求澄清章节不生成正式测试用例（Test Cases）；
- 五个生成类 Skill 的十列顺序、状态规则、模块分割行和同源双交付通过检查；
- 正式服章节缺四项门禁时只有 L0 只读测试用例（Test Cases）；
- 审计结论能回指需求、接口和测试用例（Test Case）ID。

### 15.3 第八个 Skill

- Codex 内置 Node 且无相邻 npm CLI 的首次准备；
- API-only 不下载浏览器；
- Web 交互打开可见浏览器；
- 成功、失败、待定和未执行状态分别有对应证据；
- 失败根因聚合不丢失单条测试用例（Test Case）证据；
- 每套执行前后种子状态一致；
- Excel、HTML、JSON、事件日志、截图和 Trace 一致。

### 15.4 视频

- 未剪辑版可从头播放到尾，无剪切、静音、1080p；
- 精剪版不超过 20 分钟；
- SRT 时间轴与画面一致；
- 关键 Skill 名称、测试用例（Test Cases）、不通过、待定和报告证据可读；
- 未出现个人信息、密钥、无关通知或私人窗口；
- 视频中展示的路径和文件在交付目录真实存在。

## 16. 完成条件

只有同时满足以下条件才能宣称完成：

1. 七个 Skill 都有独立的真实调用记录；
2. 五套正式测试用例（Test Cases）均由对应 Skill 生成并通过结构验证；
3. 第八个 Skill 能在干净环境自动准备并执行；
4. 通过、不通过、待定、未执行四状态均有真实且语义正确的示例；
5. 幂等缺陷被正确发现并只聚合为一个根因缺陷；
6. 优惠券过期边界被正确判定为待定而非 Bug；
7. 成功截图、失败截图、API 证据、日志、Trace 和三种报告均可打开；
8. 原始未剪辑静音录屏完整可播放；
9. 精剪静音字幕版不超过 20 分钟；
10. 所有公开材料通过隐私、链接、内容和一致性验收。

## 17. 实施顺序

实施计划应按以下依赖顺序展开：

1. 修复第八个 Skill 的无 npm CLI 启动、成功证据和待定判定；
2. 创建并测试 SkillMart 演示系统；
3. 编写 PRD v0、产品确认记录、PRD v1 和接口契约；
4. 分七次真实调用原七个 Skill 并生成归档产物；
5. 审计需求工作台生成的主测试用例（Test Cases），并按问题 ID 修订明确授权的范围；其他四套只做各自结构和执行准入验证；
6. 使用第八个 Skill 分别执行五套正式测试用例（Test Cases）；
7. 完成报告一致性、证据和缺陷聚合验收；
8. 在专用干净桌面从头连续录制完整流程；
9. 基于原始录屏生成 20 分钟以内精剪版、字幕和时间点清单；
10. 完成最终隐私和复现验收。
