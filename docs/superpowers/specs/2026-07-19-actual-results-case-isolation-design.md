# 测试用例实际结果与用例级会话隔离设计

日期：2026-07-19

## 问题证据

真实样本 `登录功能演示-正式交付/登录功能演示-测试用例.xlsx` 的测试用例表只有十列，包含“执行结果”但没有“实际结果”。其“数据与风险”Sheet 已声明“每条用例使用独立或已清理的浏览器会话”，各登录用例的前置条件也写了未登录独立会话。

执行链路没有落实这两个合同：Runner 的 `run-result.json` 保存 assertion `actual`，报告投影却只回填状态并丢弃实际观察值；整批 Web 用例只创建一个 Playwright BrowserContext，`cleanup.web` 也只是点击定位器，无法保证 Cookie、LocalStorage 和登录态隔离。

## 目标

1. `requirement-test-workbench` 生成十一列正式用例，在“优先级”和“执行结果”之间增加“实际结果”。
2. 未执行用例的实际结果固定为“尚未执行”；执行后回填去敏后的实际断言、页面状态或阻塞原因摘要。
3. Runner 同时接受历史十列和新十一列，其他现有生成 Skill 不被强制迁移。
4. 每条 Web 用例默认使用独立 BrowserContext；同一条用例内部保持连续状态。
5. 上一条用例的清理失败不能污染下一条用例。依赖登录态的刷新、重访等场景必须在同一条用例内自行建立登录态。

## 数据合同

十一列顺序为：

`用例 ID、所属模块、用例标题、验证功能点、前置条件、测试步骤、预期结果、优先级、实际结果、执行结果、备注`

十列继续作为 legacy 输入。解析器按表头识别两种标准格式，不再把十一列误判为非标准 Excel。Manifest 的 `original` 增加可选“实际结果”，从旧十列编译时补“尚未执行”。报告投影按列名定位“实际结果”“执行结果”“备注”，禁止再使用固定数组下标。

实际结果取 `run-result.json` 中每条用例的 assertions，按断言 ID 汇总 `actual`；无实际断言时根据运行状态写明“未产生业务实际结果”及原因。内容必须沿用 Runner 的凭据脱敏边界，不回填密码、令牌或密钥。

## 会话隔离合同

Runner 在用例边界提供 `beforeCase`/`afterCase` 生命周期。包含 Web 动作的用例在 `beforeCase` 获得新的 BrowserContext 和 Page；上一 Context 在切换时停止 Trace 并关闭。API-only 用例不创建额外浏览器。

浏览器进程仍只启动一次，以控制成本；隔离单位是 BrowserContext。每条用例内部的点击、登录、刷新和断言共用同一 Context。执行结束后关闭最后一个 Context 和浏览器。每个用例的 Trace 使用独立路径，避免覆盖。

## 生成 Skill 规则

需求工作台在生成 Web 用例时必须：

- 把起始状态写成可执行前置，例如“新建未登录浏览器会话并打开登录页”；
- 禁止依赖上一条用例的终态；
- 连续状态场景在本用例内完成“建态 → 动作 → 断言”；
- 区分业务清理与会话隔离，不能用“打开首页”冒充清 Cookie/Storage；
- 输出前审查每条用例是否可独立执行，以及失败后是否能恢复下一条用例的起始状态。

## 验证

以用户提供的 Word v0.1 和 Excel 为回归样本，新增测试证明：旧实现缺少实际结果、十一列被判为非标准、报告投影丢弃 actual、两条 Web 用例复用同一 Context。修改后验证十列兼容、十一列生成与回填、状态列下拉/筛选、每用例 Context 隔离、API-only 无浏览器回归、完整 Python/Node/TypeScript 测试及构建。
