# 七个测试 Skill 标准化发布与双格式输出设计

日期：2026-07-13  
状态：已确认，待用户审阅书面规格后进入实施计划

## 1. 目标

将现有七个中文测试 Skill 改造为可被 Codex 和 Claude Code 自动发现、可通过 GitHub 命令安装、可被 CC Switch 识别和管理的标准 Skill 包，同时完成以下优化：

1. 保留七个中文源文件的名称、位置、数量、职责和表达习惯。
2. 建立七个标准、独立、可单独安装的 `skills/<name>/SKILL.md` 包。
3. 解决七个 Skill 的默认路由冲突。
4. 支持一个主 Skill 与最多一个辅助 Skill 的受控组合。
5. 对五个正式用例生成 Skill 增加 Excel + HTML 双格式文件输出。
6. 在 Excel 和 HTML 中提供四状态执行结果、条件行颜色和实用交互。
7. 对七个 Skill 做内容级删重，但不得改变原意或降低行为效果。
8. 发布到现有公开仓库 `Saitamasans/testing-skills`，使用 MIT License。

## 2. 不可改变的约束

- 七个中文源文件一个不少，文件名和根目录位置不变。
- 不合并、不删除、不拆分七个 Skill。
- 单接口完整版和精炼版继续独立存在。
- 最大程度保留原中文语气、编号、表格、P0/P1/P2、待确认、合理假设等表达习惯。
- 行数目标不能凌驾于原意、功能、门禁和可用性。
- 中文源文件是内容唯一真源，不人工维护两套正文。
- 用户确认书面规格前不修改七个 Skill。
- 用户再次确认前不推送 GitHub、不创建 Release。

### 3.1 构建规则

1. 根目录七个中文文件是唯一人工维护的 Skill 正文。
2. `skills/*/SKILL.md` 由构建工具自动生成并提交到 Git。
3. 自动生成文件标明“自动生成，请勿直接修改”。
4. 每个标准包必须自包含，使用 `--path skills/<name>` 安装时不能依赖包外正文。
5. 五个用例生成包携带 Excel/HTML 输出脚本；用例审计和需求澄清不携带测试执行状态组件。
6. 通用渲染器和主题在 `tooling/` 中维护，由构建工具复制到五个标准包，避免人工维护五份实现。
7. 用户从 GitHub 安装时不需要先运行构建命令。
8. GitHub Actions 检查生成包是否与中文真源同步。

## 4. 七个 Skill 的主路由

每次先选定一个主 Skill：

| 优先条件 | 主 Skill |
|---|---|
| 用户提供已有测试用例并要求审计、评审、查漏或修订建议 | 用例审计 |
| 用户明确只澄清需求、判断能否开测、暂时不要测试用例 | 需求澄清 |
| 用户要求正式服、线上、生产环境或上线后验证 | 正式服验证 |
| 用户显式要求精炼版、快速版、短版或低上下文 | 单接口精炼版 |
| 用户提供一个接口资料并要求接口测试 | 单接口完整版 |
| 用户提供多个接口、调用链或上下游关系 | 多接口链路 |
| 用户提供 PRD、用户故事或需求变更并要求测试设计、测试点或用例 | 需求工作台 |

### 4.1 description 边界

- 单接口完整版：普通单接口任务的默认入口。
- 单接口精炼版：仅由显式短版口令触发。
- 多接口链路：两个及以上接口、调用链或字段依赖；默认面向测试和预发。
- 需求工作台：以需求为输入，目标是测试设计或用例资产。
- 需求澄清：明确只澄清、暂不写用例。
- 用例审计：输入重点是已有测试用例。
- 正式服验证：核心目标是在正式服、线上或生产环境执行验证。

## 5. 主 Skill 与辅助 Skill

一个任务只能有一个主 Skill，最多建议一个辅助 Skill。

| 主 Skill | 可建议的辅助 Skill | 使用场景 |
|---|---|---|
| 正式服验证 | 单接口完整版或精炼版 | 线上验证一个接口 |
| 正式服验证 | 多接口链路 | 线上验证多个接口链路 |
| 用例审计 | 正式服验证 | 审计正式服执行用例是否安全 |
| 需求工作台 | 正式服验证 | PRD 的交付目标是上线后验证 |
| 需求工作台 | 单接口或多接口 | PRD 带有明确接口契约，需要接口专项增强 |

辅助 Skill 只补充专项分析，不改变主 Skill 门禁，不单独生成第二套聊天结果、Excel 或 HTML。最终输出结构由主 Skill 决定。

### 5.1 用户确认门禁

检测到辅助 Skill 时，先展示主 Skill、辅助 Skill、分工和最终输出规则，然后暂停等待用户确认。

用户确认前：

- 不加载辅助 Skill；
- 不执行辅助分析；
- 不生成最终文件；
- 不把沉默视为同意。

用户拒绝时，主 Skill 继续执行，并明确说明缺少辅助分析可能造成的覆盖限制。

### 5.2 调用透明度

聊天开头、Excel 总览和 HTML 总览均记录：

- 主 Skill；
- 辅助 Skill；
- 各自职责；
- 用户确认结果；
- 辅助 Skill 被拒绝时的覆盖限制。

## 6. 内容级优化

优化优先级：

```text
原意不变
> 功能和行为不退化
> 关键门禁不丢失
> 原表达习惯保留
> 行数压缩
```

目标行数为适度放宽目标，不是机械硬限制：

| Skill | 目标行数 |
|---|---:|
| 单接口完整版 | 约 600 行以内 |
| 单接口精炼版 | 约 264 行以内 |
| 多接口精炼版 | 约 420 行以内 |
| 需求工作台 | 约 420 行以内 |
| 正式服验证 | 约 360 行以内 |
| 用例审计 | 约 360 行以内 |
| 需求澄清 | 约 336 行以内 |

如果不降低效果时无法达到目标，可以超过目标，但必须说明保留原因。

### 6.1 逐个优化方向

- 单接口完整版：合并文件触发、L0/L1 降级、禁止事项和最终自检的重复定义，保留全部独有分析能力和完整 Sheet 契约。
- 单接口精炼版：改为线性正向配方，保持独立可用，删除重复 Sheet 解释和同义禁止项。
- 多接口精炼版：保留当前主体，仅补充路由、主辅调用、双格式输出和四状态规则。
- 需求工作台：合并模式选择表与触发口令表的重复，保留原口令、全部模式和显式全量门禁。
- 正式服验证：生产四项门禁只保留一个权威定义；默认模板只展示 L0 只读示例；写操作示例放入全部门禁通过后的条件区。
- 用例审计：证据铁律前移，保留四级审计和默认只审计边界，合并重复 Sheet 字段与自检。
- 需求澄清：修正标题跳级，合并重复边界和复审规则，保留 `C001/Q001/N001` 及三个业务模板。

### 6.2 中文表达保护

检查病句、双重否定、主语缺失、前后指代、逻辑顺序和类似“用户未确认前”的不自然表达。统一使用“用户确认前”或“用户尚未确认时”。不把原中文测试术语替换为陌生工程术语。

### 6.3 能力保留清单

每个 Skill 修改前后核对：触发能力、核心场景、阻断门禁、输出字段、业务模板、原口令和关键行为回归。压缩不得导致 Skill 没那么管用或没那么好用。

## 7. Excel 与 HTML 双格式输出

适用于：单接口完整版、单接口精炼版、多接口链路、需求工作台和正式服验证。

用户明确要求生成文件时，同时输出 `.xlsx` 和 `.html`。用户只要求聊天内容时不自动生成文件。用例审计和需求澄清不使用测试执行状态组件。

### 7.1 统一十列与四状态

正式用例继续使用：

```text
用例 ID | 所属模块 | 用例标题 | 验证功能点 | 前置条件 | 测试步骤 | 预期结果 | 优先级 | 执行结果 | 备注
```

执行结果包含：

| 状态 | 含义 |
|---|---|
| 未执行 | 尚未开始执行 |
| 通过 | 已执行，符合已确认口径 |
| 不通过 | 已执行，明确违反已确认口径 |
| 待定 | 已执行，但需求、产品、研发或测试口径存在歧义，暂时不能定性为缺陷 |

选择“待定”时不强制填写备注，由测试人员自行决定。

### 7.2 Excel/WPS

- 正式用例行的执行结果单元格使用四状态数据验证下拉。
- 模块分割行不添加下拉框。
- 不通过：整行淡红，最高优先级。
- 待定：整行淡灰。
- 通过和未执行：不覆盖原模块色和优先级色。
- 支持 Excel 2016 及以上和当前主流 WPS Office。
- 不使用宏或 Microsoft 365 专属函数。

### 7.3 HTML

HTML 为单文件、离线可用，不依赖服务器、CDN 或网络请求。包含：

- 关键词搜索；
- 所属模块筛选；
- 优先级筛选；
- 执行状态筛选；
- 冻结表头；
- 四状态统计；
- 四状态下拉；
- 条件行颜色；
- 浏览器本地自动保存。

每份 HTML 使用“Skill + 项目/模块 + 生成时间 + 内容摘要”生成独立存储标识。浏览器允许本地存储时，刷新或重新打开后恢复状态；浏览器禁止时显示明确警告，页面仍可使用，不伪称已保存。

### 7.4 一致性

Excel 和 HTML 由同一份结构化用例数据一次生成。自动验证用例数量、ID、十列、初始状态、模块顺序和优先级一致。生成后两份文件各自独立编辑，不做双向同步。

## 8. GitHub、安装与 CC Switch

发布到现有公开仓库 `Saitamasans/testing-skills`，采用 MIT License。

README 中文为主，附简短英文名称和英文 description，包含七个 Skill 功能表、逐个安装、一键安装全部、兼容性、主辅调用、双格式输出、升级、卸载和检查更新。

逐个安装示例：

```bash
npx skills add Saitamasans/testing-skills --path skills/single-api-test-full
```

所有 README 命令发布前必须真实执行验证。若 `npx skills` 不支持稳定的一键多 Skill 安装，则保留官方逐个安装，并提供经过验证的仓库级安装脚本，不写未经验证的命令。

兼容验收：

- Codex：必须可安装、发现和调用。
- Claude Code：必须可安装、发现和调用。
- CC Switch：必须识别七个 Skill，并正确显示名称、description 和平台状态。
- Gemini、OpenCode、Hermes：说明兼容情况，不作为本轮阻塞验收项。

## 9. GitHub Actions

每次 push 或 Pull Request 自动执行：

1. 构建七个标准包；
2. 检查生成结果与中文真源同步；
3. 校验目录、frontmatter、name 和 description；
4. 校验路由边界；
5. 校验关键能力保留清单；
6. 扫描占位符、病句规则和内部临时文件；
7. 测试五个 Skill 的 Excel/HTML 生成器；
8. 检查 Excel 下拉与条件格式；
9. 浏览器测试 HTML 搜索、筛选、统计、颜色和本地保存；
10. 验证 README 安装路径和安装冒烟；
11. 报告行数超标，但不驱动机械删减。

## 10. 失败处理

- 任一中文真源缺失或 frontmatter 无效时停止构建。
- 生成包不同步时 CI 失败。
- 五个生成包缺少输出脚本时 CI 失败。
- Excel 或 HTML 任一个失败时不得声称双格式成功。
- README 命令未通过验证时不得发布。
- CC Switch 未识别七个 Skill 时不得标记兼容完成。
- 构建使用临时目录；全部验证通过后再替换生成包，避免半成品。

## 11. 验证与验收

### 11.1 路由回归

覆盖普通单接口、显式短版、多接口、PRD 测试设计、只澄清、已有用例审计、正式服主路由、辅助 Skill 确认、拒绝辅助 Skill 和组合调用单一输出。

### 11.2 Excel 验收

五个 Skill 分别生成样例，检查 Excel/WPS 打开、十列、四状态下拉、模块分割行、默认未执行、条件色、筛选、冻结、列宽、换行和字体。

### 11.3 HTML 验收

使用 Edge/Chrome 检查离线打开、搜索、筛选、冻结、统计、状态、条件色、本地保存、报告隔离和无外部请求。

### 11.4 发布验收

- 七个中文源文件名称和位置不变；
- 七个标准安装包存在；
- 不合并、不删除、不拆分；
- 路由和主辅确认通过；
- 五个生成类 Skill 双格式通过；
- 四状态语义和颜色通过；
- Codex、Claude Code 和 CC Switch 验收通过；
- GitHub Actions 全绿；
- README 命令真实可执行；
- MIT License 存在；
- 工作区干净。

## 12. 实施节奏

实施采用一次批量修改、一次统一回归、一次最终审查。不得再次对七个纯文本 Skill 执行多轮无必要的串行审查。正式实施前必须先由用户审阅本设计文档，再创建实施计划。
