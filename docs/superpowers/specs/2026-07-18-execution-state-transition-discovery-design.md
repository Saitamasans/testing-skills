# 第八个 Skill 状态迁移探测与核心链路覆盖设计

日期：2026-07-18

范围：`web-api-test-execution-evidence` Skill 源文档、生成包、安装包和契约测试

明确不包含：固定 Runner 协议、Runner 源码、现有测试用例和既有执行产物的修改

## 一、问题与根因

当前黑盒 Web 流程只允许对初始目标页执行只读 `discover-web`。当测试用例需要通过点击、提交或键盘动作进入尚未探测的新页面或新状态时，现有规则没有定义后续探测流程。

这会产生错误降级：目标状态的定位器和断言契约尚未确认，规划器便把整条业务用例替换为 `execution.blocked`。即使起始页的迁移动作已经唯一、可见且经过确认，也不会执行。最终只执行准备或输入用例，却可能被表述成完整业务测试已经完成。

根因不是 Runner 不支持点击。固定 Runner 已支持 `web.click`、`web.fill`、`web.assert` 和 `url-contains=`。根因是 Skill 缺少以下四个规划契约：

1. 页面和业务状态迁移的显式建模；
2. 测试用例动作到 manifest 动作的守恒审查；
3. 核心业务目标的端到端覆盖门禁；
4. 部分执行结果的准确命名和交付措辞门禁。

## 二、设计目标

修复后，第八个 Skill 必须做到：

1. 在生成 manifest 前识别跨页面、跨路由、弹窗、新标签页、登录回调、下载和异步结果状态等迁移；
2. 目标状态尚未探测时，优先进入分阶段状态迁移探测，而不是直接阻塞整条用例；
3. 不猜测目标页定位器，不查看源码，不调用内部 API；
4. 不把探测性迁移动作回填成正式测试通过；
5. 核心业务结果没有完整可执行路径时，不允许进入 E4，不允许称为“全部执行”或“测试完成”；
6. 对类似缺口在第一次确认门禁前暴露，避免用户在录屏或正式执行后才发现没有触达业务目标；
7. 保持固定 Runner 1.1.0 不变。

## 三、状态图与动作守恒审查

### 3.1 状态图

每条 Web 测试用例在规划前拆成：

- 起始状态：当前已经只读探测的页面或业务状态；
- 迁移动作：点击、提交、Enter、选择、上传、打开新标签页等；
- 目标状态：结果页、工作台、确认页、弹窗、下载完成态等；
- 终态业务断言：用于证明业务目标已经达到的可观察结果。

相同目标状态可复用已确认的同源 discovery；不同 origin、不同页面模板、不同登录态或不同主要内容区域不得静默复用。

### 3.2 动作守恒矩阵

第一次确认门禁前必须逐用例输出动作守恒矩阵。测试步骤中的每个业务动作必须处于以下一种状态：

- `mapped`：已映射为具体 Runner 动作；
- `transition_discovery_required`：起始定位器已确认，但目标状态尚待分阶段探测；
- `blocked`：存在明确、不可在当前权限和能力内消除的能力缺口；
- `manual_required`：验证码、扫码、MFA 或只能人工完成的操作。

禁止把包含已确认迁移动作的整条用例直接压缩成一个笼统的 `execution.blocked`，除非先证明状态迁移探测也不可执行，并展示具体原因。

## 四、分阶段状态迁移探测

### 4.1 触发条件

同时满足以下条件时，进入 `transition_discovery_required`：

1. 用例包含进入新页面或新业务状态的动作；
2. 起始页动作定位器已经通过只读 discovery 唯一确认；
3. 目标状态的定位器或终态断言尚未通过只读 discovery 确认；
4. 迁移动作风险为 R0/R1，且没有验证码、扫码、MFA 或不可逆副作用。

### 4.2 迁移探测方案

Skill 生成独立的“状态迁移探测预览”，只能使用现有测试用例中的动作和预期，不能生成新测试用例。预览必须展示：

- 来源测试用例 ID；
- 起始状态与目标状态；
- 将执行的前置动作和迁移动作；
- 已确认定位器、匹配数、可见数和置信度；
- 一个来自原用例预期的最小迁移断言，例如 URL 发生变化、URL 包含现有测试数据，或已确认的状态文本出现；
- 风险等级、动作数、环境变量和独立审批 hash；
- 明确声明“本次仅用于到达并探测目标状态，不产生正式测试结论，不回填 Excel/HTML”。

用户单独确认后，使用固定 Runner 的现有 `web.goto`、`web.fill`、`web.click`、`web.wait` 和 `web.assert` 执行迁移前缀。输出必须隔离在 discovery 目录，不得并入正式 `run-result.json`。

迁移成功后，从真实执行证据中取得实际目标 URL 或活动页面，再对该目标执行只读 `discover-web`。页面候选不唯一、目标状态未达到或发生反自动化时停止并要求人工处理，不猜定位器。

### 4.3 正式 manifest

目标状态 discovery 由用户确认后，才允许把迁移动作、目标页定位器和终态业务断言写入正式 `case_plans`。正式 manifest 必须重新生成 discovery/proposal hash、manifest hash、动作数和断言预览，并重新经过第二次确认门禁。

迁移探测的通过不等于正式用例通过。只有正式 manifest 完整执行且所有显式业务断言通过，测试用例才能判定为通过。

## 五、核心业务目标覆盖门禁

### 5.1 核心路径识别

根据用例标题、验证功能点、优先级和预期结果识别用户要求验证的核心业务目标。对于搜索场景，核心路径至少包含“输入关键词 → 触发搜索 → 到达结果状态 → 观察结果业务断言”；只验证首页和输入不算覆盖搜索目标。

### 5.2 E4 门禁

生成正式 manifest 前计算并展示：

- 总测试用例数；
- 完整可执行用例数；
- blocked/manual 数；
- 核心业务路径数；
- 完整可执行核心路径数；
- 每个核心目标的终态断言覆盖情况。

用户要求完整执行时，只要任一核心目标没有至少一条完整可执行路径，就不能进入 E4，不能生成可供正式执行确认的最终 manifest。必须先完成状态迁移探测、修复能力缺口，或取得用户对“仅部分执行/准备度验证”的明确降级确认。

普通的“确认执行”不得被解释为接受核心业务目标降级。降级确认必须逐项列出未覆盖目标及影响。

## 六、相似问题预防审查

第一次确认门禁前必须执行以下审查：

1. **动作缺失审查**：测试步骤中的点击、提交、Enter、选择、上传等是否全部映射；
2. **终态缺失审查**：每个核心目标是否有目标状态和业务断言；
3. **状态迁移审查**：是否存在未探测的新页面、弹窗、新标签页、回调或下载态；
4. **阻塞传播审查**：目标状态未知是否错误地抹掉了可执行的迁移探测；
5. **替代动作审查**：不得用点击冒充 Enter、用 URL 打开冒充真实提交、用动作完成冒充业务通过；
6. **部分执行审查**：执行集合是否只剩准备、首页或输入用例，却仍被描述为核心流程测试；
7. **风险审查**：高风险提交不得因追求覆盖而进入迁移探测；
8. **报告措辞审查**：最终结论是否准确区分完整执行、部分执行、准备度验证和 blocked。

审查发现问题时必须在第一次门禁前纠正；不能把已知的核心覆盖缺口推迟到录屏或正式执行后再说明。

## 七、报告与用户沟通契约

最终结论必须同时报告“执行数量”和“业务目标覆盖”。

当核心用例 blocked 时：

- 禁止使用“全部测试完成”“搜索测试完成”“核心流程通过”等措辞；
- 必须使用“部分执行”“准备度验证结果”或“核心流程未执行”；
- 必须首先列出未触达的核心动作和业务结果；
- 通过的准备类用例不能抵消核心业务目标未覆盖。

录屏、Excel、HTML 和 JSON 一致，只证明产物一致，不证明核心业务目标已覆盖。

## 八、测试设计

### 8.1 RED 基线回归

新增契约测试，使用本次百度场景作为最小失败样例：

- `INPUT-001` 可以输入并断言不自动导航；
- `SEARCH-001` 要求点击“百度一下”并观察结果状态；
- 旧规则把 `SEARCH-001` 压缩成 `execution.blocked`，仍允许生成“2+8”正式执行预览；
- 测试必须在修改前失败，证明当前 Skill 缺少状态迁移和核心覆盖门禁。

### 8.2 GREEN 契约测试

修改后测试必须验证源 Skill 和生成 Skill 同时包含：

- `transition_discovery_required`；
- 状态图和动作守恒矩阵；
- 核心路径完整可执行覆盖门禁；
- 迁移探测不得回填正式结果；
- 核心用例 blocked 时禁止完整执行措辞；
- 目标状态 discovery 后必须重新生成和确认正式 manifest。

### 8.3 相似场景回归

增加以下静态压力场景或契约样例：

1. 搜索首页 → 搜索结果页；
2. 登录页 → 工作台；
3. SPA 页面 → 弹窗或异步结果区域；
4. 当前页 → 新标签页；
5. 提交页 → 下载或确认页；
6. 支付、权限、库存等 R2/R3 动作必须阻止自动迁移探测；
7. Enter 不受支持时不得用点击替代，但必须准确报告对应核心路径未覆盖。

## 十、验收标准

修复完成必须同时满足：

1. 百度场景在第一次确认门禁前被识别为“核心搜索路径尚未完整可执行”；
2. 系统提出状态迁移探测，而不是直接把点击搜索整条用例降为 blocked；
3. 用户未确认迁移探测前不点击；
4. 目标结果页未只读探测前不猜定位器；
5. 只执行首页和输入时，结果只能称为部分执行或准备度验证；
6. R2/R3 和人工验证场景不会被自动状态迁移探测绕过；
7. 源文件、生成包和安装副本关键契约一致；
8. 新增回归测试经历 RED → GREEN，完整测试、构建和验证命令通过。
