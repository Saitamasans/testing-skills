# Web/API 测试用例自动执行与证据回填 Skill 设计

日期：2026-07-14

Skill package：`web-api-test-execution-evidence`

Runner package：`@saitamasans/testing-runner`

## 1. 背景与目标

现有 `testing-skills` 已覆盖需求澄清、需求测试设计、单接口、多接口链路、正式服验证和已有用例质量审计，但尚未形成“正式用例 → 自动执行 → 原始证据 → 准确判定 → 双格式回填”的执行闭环。

本项目新增一个独立 Skill，通过版本化的本地 Runner 执行 Web/API 混合测试，并可选执行只读数据库核对。它不追求以 AI 临场操作覆盖所有用例，而以用例准确、执行准确、断言准确和报告准确为最高目标。

核心优先级：

```text
准确性
> 可追溯性
> 安全性
> 自动化覆盖率
> 执行速度
```

宁可阻塞、少执行或转人工，也不得猜测步骤、猜测断言、伪造通过或把执行器异常冒充研发缺陷。

## 2. 独立定位与现有七个 Skill 的关系

`web-api-test-execution-evidence` 是独立的第八个 Skill：

- 不要求安装或调用现有七个 Skill；
- 不默认重新生成、改写或审计用户提供的测试用例；
- 可独立处理测试人员手写或其他工具导出的 Excel 用例；
- 原生兼容 `Saitamasans/testing-skills` 的十列顺序、四状态和 Excel/HTML 报告协议；
- 现有七个 Skill 不依赖本 Skill，继续独立工作。

输入检测分为：

- **原生增强模式**：识别到 `testing-skills` 的 `report.json` 和调用元数据，直接复用已确认门禁、字段和报告协议；
- **兼容模式**：识别到外部 Excel，先做字段映射、完整性检查和用户确认。

兼容模式显示非阻塞提醒：使用 `Saitamasans/testing-skills` 完成需求澄清、用例生成或质量审计，可获得更完整的门禁信息、断言依据和双格式报告；当前外部用例仍可继续处理。

外部用例质量不足时，可建议 `test-case-quality-audit` 作为最多一个辅助 Skill。调用前必须展示主/辅助 Skill 与分工并等待用户确认；拒绝后继续兼容模式并说明覆盖限制。

## 3. 第一版输入范围

第一版接收：

1. 标准 `report.json`；
2. 标准十列 `.xlsx`；
3. 任意列结构的非标准 `.xlsx`。

第一版不直接接收 HTML、CSV、Markdown 或纯自然语言作为正式执行输入。HTML 是交互执行报告，不是用例真源；其他格式可在后续通过独立输入适配器扩展。

### 3.1 标准十列

```text
用例 ID | 所属模块 | 用例标题 | 验证功能点 | 前置条件 |
测试步骤 | 预期结果 | 优先级 | 执行结果 | 备注
```

### 3.2 非标准 Excel 映射

非标准 Excel 必须经过映射预览和用户确认，不允许静默转换，即使字段匹配置信度很高。

映射报告至少包含：

- 原 Sheet、原列名和样例值；
- 建议映射的统一字段；
- 匹配依据与置信度；
- 缺失字段、重复映射、冲突映射；
- 未识别的扩展列；
- 合并字段的拆分预览；
- 转换后的标准十列样例。

规则：

- 缺少执行结果时可默认 `未执行`；
- 备注允许为空；
- 缺少用例 ID 时生成 `EXT-<批次>-<序号>` 临时 ID，并保留来源映射；
- 额外字段进入不可丢失的原始数据快照与扩展元数据；
- 缺少测试步骤或预期结果时禁止进入自动执行；
- 一个原字段需要拆成步骤与预期时，必须展示拆分预览并等待确认；
- 映射确认记录、源文件哈希和映射版本必须归档。

## 4. 执行前准备度门禁

准备度检查是第零道门禁，早于准确性编译和 Runner 启动。

### 4.1 用户最低准备项

必须项：

- `report.json` 或 Excel 测试用例；
- Web/API 的实际目标地址；
- 可执行测试步骤；
- 明确预期结果；
- 对本批测试范围的授权与确认。

按命中场景提供：

- Web 登录账号、密码或登录方式；
- API Token、Header 或鉴权获取方法；
- 只读数据库连接信息；
- 测试数据或测试数据创建方法；
- 文件上传样例；
- SSO、验证码、扫码或 MFA 的人工接管；
- 测试数据清理接口或人工清理责任人。

用户不需要提前准备 Playwright 脚本、页面元素定位、机器执行清单、Trace、截图或报告模板。未命中数据库断言时不询问数据库凭据；公开页面不询问登录账号；无写操作不强制询问清理接口。

### 4.2 准备度等级

| 等级 | 条件 | 行为 |
|---|---|---|
| E0 | 没有正式用例，只有测试意图 | 不执行；建议使用现有七个 Skill 或提供正式用例 |
| E1 | 有用例，缺少目标、账号或鉴权 | 输出具体缺失清单，不启动 Runner |
| E2 | 可编译草案，部分数据、断言或清理方式不完整 | 生成执行草案与阻塞项，等待补充 |
| E3 | 用例、目标、凭据引用、数据、断言和清理方式齐全 | 展示执行预览，等待用户确认 |
| E4 | 执行清单已确认并锁定 | 允许交互执行或受控 CI 执行 |

提示必须列出已具备项、阻塞项、可选增强、阻塞原因和可直接复制的补充示例，不得只说“信息不足”。

### 4.3 本机运行条件

Skill 自动检查 Runner 兼容版本、Node、Playwright 浏览器、网络连通性和可选数据库驱动。缺失依赖时先展示安装内容、来源、版本和影响，等待用户确认后再安装；不得擅自安装软件或驱动。

## 5. Monorepo 与独立版本

Skill 与 Runner 位于同一个 `testing-skills` Monorepo，方便同步协议；Runner 作为独立 npm 包测试和发布。

```text
testing-skills/
├─ skills/
│  └─ web-api-test-execution-evidence/
│     ├─ SKILL.md
│     ├─ agents/openai.yaml
│     └─ references/
├─ packages/
│  └─ testing-runner/
│     ├─ src/
│     ├─ tests/
│     ├─ package.json
│     └─ bin/
├─ schemas/
│  ├─ report.schema.json
│  ├─ execution-profile.schema.json
│  ├─ run-manifest.schema.json
│  ├─ approval.schema.json
│  └─ run-result.schema.json
└─ knowledge/
   ├─ technical-rules.json
   └─ high-risk-heuristics.json
```

Skill 锁定兼容的 Runner 版本和协议版本。Runner 修复、规则变更和行为变化必须独立记录版本，不允许使用未锁定的 `latest` 在 CI 中执行。

目标命令界面：

```bash
npx @saitamasans/testing-runner@1.0.0 plan --input report.json --profile execution-profile.json
npx @saitamasans/testing-runner@1.0.0 run --manifest run-manifest.json --approval approval.json
```

## 6. 执行器选择

采用“统一核心 + 可选适配器”。

### 6.1 核心执行器

Runner 使用 TypeScript + Playwright Test：

- Web 使用浏览器上下文、稳定定位、条件等待、截图和 Trace；
- API 使用 Playwright `APIRequestContext`，支持 Cookie、鉴权、变量提取和 Web/API 混合链路；
- 同一业务用例可在 API 创建数据、Web 完成操作、API 查询结果、数据库只读核对；
- 统一使用 Playwright 的 Fixture、超时、附件和测试生命周期；
- Runner 解释声明式清单，不执行 AI 临时生成的任意 Shell 或脚本。

### 6.2 可选适配器

- 只读数据库：MySQL/PostgreSQL 等驱动按需安装；
- Postman Collection：后续接 Newman；
- OpenAPI 契约/异常测试：后续接 Schemathesis。

Runner 核心不强迫用户提供 Postman 或 OpenAPI 资产。可选适配器的结果必须转换为统一事件和断言协议。

### 6.3 白名单动作

第一版只解释声明式动作，例如：

```text
web.goto
web.fill
web.click
web.select
web.wait
web.assert
api.request
api.extract
api.assert
db.select
cleanup.api
cleanup.web
```

未进入白名单的动作不能运行。每一步必须关联用例 ID、来源步骤、目标别名、输入引用、风险级别、断言和证据策略。

## 7. 执行产物与数据流

```text
report.json / Excel
→ 字段映射与准备度检查
→ execution-profile.json
→ run-manifest.json
→ 执行预览与用户确认
→ approval.json
→ Runner
→ run-events.jsonl
→ run-result.json
→ evidence/
→ result.xlsx + result.html
```

产物职责：

| 文件 | 用途 |
|---|---|
| `report.json` | 标准十列和报告元数据真源 |
| `execution-profile.json` | 目标别名、凭据引用、风险和策略，不含秘密值 |
| `run-manifest.json` | 已解析的机器执行清单 |
| `approval.json` | 用户确认范围、清单哈希、目标摘要和有效期 |
| `run-events.jsonl` | 追加写入的原始执行事件，不允许回写覆盖 |
| `run-result.json` | 断言、运行状态、用例结果和证据索引 |
| `evidence/` | 截图、Trace、日志、脱敏请求响应、只读查询和清理记录 |
| `result.xlsx/.html` | 同源回填后的人工执行载体 |

## 8. 机器执行清单

`run-manifest.json` 至少包含：

- 协议版本、Runner 兼容范围；
- 源文件、源哈希、映射版本；
- 用例 ID 和原十列快照；
- Web/API/数据库目标别名；
- 凭据环境变量引用；
- 前置条件和数据 Fixture；
- 白名单执行步骤；
- 必须断言与辅助断言；
- 断言来源、知识规则 ID、置信度和判定策略；
- 数据清理步骤；
- 证据策略；
- 风险分级；
- 定位器版本和修复历史；
- 执行顺序、超时和可重试条件。

Skill 生成清单后只做 Schema、来源和门禁校验。用户确认后计算 SHA-256；Runner 执行前验证 `approval.json` 中的哈希、目标范围和有效期。确认后清单发生任何变化都必须重新展示并确认。

## 9. 执行授权与 CI

### 9.1 交互模式

执行前展示：

- 目标 Web/API/数据库地址；
- 账号和凭据引用，不展示秘密值；
- 用例数量与执行顺序；
- 读写动作、外部副作用和风险；
- 测试数据、标记与清理方式；
- 必须断言、知识库辅助断言；
- 预计耗时、证据范围和阻塞项。

用户一次确认后整批运行，不对每个普通点击重复确认。确认只覆盖预览中列出的清单和目标。

### 9.2 受控 CI 模式

CI 只运行已人工确认、锁定并仍在有效期内的清单：

- 校验清单哈希、目标允许列表、Runner 版本和规则库版本；
- 只从 CI Secret 读取凭据；
- 禁止 AI 在运行期间添加步骤、断言或目标；
- 默认只允许 R0/R1；更高风险不得依赖普通 CI 定时任务；
- 哈希、目标或策略变化时拒绝执行并要求重新确认。

## 10. 不判断环境，只判断实际风险

Skill 不根据域名、数据库名、部署名称或账号名称猜测正式服/测试服。用户声明的环境只作为说明，不是安全结论。

风险基于：

- 实际 Web/API/数据库目标；
- 账号权限；
- 数据是否真实、共享、敏感；
- 操作是否写入、可识别、可清理；
- 是否影响资产、配置、外部系统或第三方人员；
- 是否可回滚和停止。

| 等级 | 示例 |
|---|---|
| R0 | 页面查看、API 查询、数据库只读核对 |
| R1 | 可识别、可限制、可清理的测试数据写入 |
| R2 | 共享/真实/敏感数据、高权限账号、混合目标 |
| R3 | 扣资产、发奖、改配置、外部通知、不可逆操作 |

“测试域名 + 真实数据库”按 R2/R3 处理，不争论环境名称。R2/R3 必须在预览中逐项列明；R3 不得被一个笼统的“全部运行”授权覆盖，且不能进入普通 CI 自动回归。

## 11. 凭据与登录

凭据优先级：

1. 用户本次明确提供并授权使用的 Web/API/数据库凭据；
2. 当前执行配置中的环境变量或 Secret 引用；
3. 用户明确同意复用且目标匹配、未过期的登录状态；
4. SSO、验证码、扫码或 MFA 的人工接管。

规则：

- 密码、Token、Cookie、连接串不进入用例、执行清单、日志和报告；
- 用户直接提供的秘密默认仅在本次会话使用，不长期保存；
- 是否保存 Playwright `storageState` 必须单独询问；
- 请求响应、截图、控制台日志和数据库结果在落盘前脱敏；
- 脱敏规则必须覆盖常见认证头、Cookie、手机号、邮箱、证件和自定义敏感字段；
- 数据库连接只允许只读账号；Runner 同时执行 SQL 语法与连接权限双重检查。

## 12. 数据库只读边界

数据库是可选断言来源，不是必填能力：

- 只允许单条 `SELECT`；
- 禁止 `INSERT`、`UPDATE`、`DELETE`、DDL、存储过程、文件写入和多语句；
- 不使用数据库直接创建或清理测试数据；
- 查询必须声明用途、关联用例和返回字段；
- 大结果集设定上限，超限停止并提示；
- 查询结果脱敏后才能进入证据；
- 未提供数据库信息时，清楚说明本次只验证 Web/API，不降低已有断言标准。

## 13. Web 定位与受控修复

定位优先级：

```text
data-testid
→ role + accessible name
→ label
→ 稳定业务文本
→ 稳定 CSS
```

默认禁止坐标定位和脆弱的层级选择器。首次执行前进行页面侦察并生成定位清单。动态列表优先通过业务 ID、名称或行内语义定位。

定位失败时：

1. 停止当前用例，不继续猜测点击；
2. 保存旧定位、DOM/可访问性证据和失败现场；
3. AI 可生成新的语义定位建议；
4. 展示旧/新定位、匹配元素和置信度；
5. 用户确认后形成新定位版本并重跑；
6. 不用重跑成功覆盖第一次失败记录。

## 14. 断言准确性与多层冲突

断言来源优先级：

1. 原用例中已确认的预期；
2. 项目规则和接口契约；
3. 版本化技术标准；
4. 受控的通用高风险行业经验。

页面、API、数据库分别验证用户可见行为、服务响应和持久化状态，没有天然固定优先级。执行清单提前标记必须断言；任一必须断言失败，整条用例判为 `不通过`。未在用例或清单中要求的跨层一致性冲突，记录实际结果并标记 `待定`，交给人工审核。

AI 不得修改原预期来适配当前实现，也不得把实现现状当需求。

### 14.1 知识库

第一版包含：

- HTTP、接口契约、鉴权、安全、数据一致性技术规则；
- 少量通用高风险经验：幂等、重复提交、重复扣减、库存、权限、跨用户数据隔离。

暂不包含支付、电商、活动、奖励等完整业务知识包。

每条规则包含：

```text
rule_id
version
applicability
assertion
source
exceptions
confidence
verdict_policy
```

`verdict_policy` 分为 `auto` 和 `candidate_only`。项目规则、接口契约和白名单技术规则可自动判定；行业经验默认只生成候选判定，只有明确适用、置信度高、在预览中展示并获准的规则才可自动判定。行业经验断言默认不超过本批自动断言的 10%；超过时必须说明原因并重新确认。

报告记录判定来源、判定方式、规则 ID、规则版本、置信度和是否需人工复核。

## 15. 状态模型

业务用例结果保持四状态：

```text
未执行 | 通过 | 不通过 | 待定
```

Runner 另设运行状态：

```text
planned | running | completed | blocked |
executor_error | infrastructure_error | manual_required
```

映射规则：

- 所有必须断言通过：`通过`；
- 任一必须断言明确失败：`不通过`；
- 已执行但需求、产品、研发或测试口径存在歧义：`待定`；
- 输入阻塞、执行器异常、基础设施异常：不产生业务失败判定，保留原用例结果并记录运行状态；
- 环境或执行器异常不得计入研发 Bug；
- `待定`不得被滥用于登录失败、网络失败或定位失败。

## 16. 测试数据生命周期

采用“业务渠道创建、业务渠道清理”：

- 每批生成唯一 `run_id`；
- 创建数据带可识别的业务标记；
- 优先通过 API 创建，必要时通过 Web UI 创建；
- 完成后按依赖逆序通过业务 API/UI 清理；
- 数据库始终只读，不参与清理；
- 清理失败时保留数据 ID、创建时间、所属用例、目标和失败原因；
- 输出人工清理清单，不得宣称已清理；
- 已有共享数据默认不做修改，除非清单中明确授权。

## 17. 重试规则

- 业务断言失败不自动重试；
- 页面加载超时、网络闪断、浏览器崩溃等执行器/基础设施异常允许重试一次；
- 异步业务状态使用明确条件等待和超时，不用反复重跑代替；
- 第一次失败与重试证据全部保留；
- 第一次异常、第二次成功时，报告仍记录发生过执行异常；
- 重试后仍失败时使用运行状态，不冒充业务 `不通过`。

## 18. 证据策略

通过用例保存：

- 执行摘要；
- 关键断言；
- 必要截图；
- 脱敏请求摘要；
- 清理结果。

不通过、待定和执行异常保存：

- 完整 Playwright Trace；
- 失败/冲突截图；
- 脱敏请求与响应；
- 控制台和 Runner 日志；
- 只读数据库结果；
- 重试记录；
- 清理记录或人工清理清单；
- 知识规则和判定来源。

证据按 `run_id/<case_id>/` 归档，并建立 SHA-256 索引。第一次失败证据只追加不覆盖。

## 19. 报告回填

同一份 `run-result.json` 同时驱动 Excel 和 HTML，禁止分别计算结果。

回填内容：

- 十列中的执行结果；
- 备注中的运行状态、自动判定标识、重试、证据相对链接和人工复核说明；
- Excel/HTML 总览中的 Runner、协议、清单哈希、规则库版本和执行摘要；
- 四状态统计与执行异常统计分开；
- 清理清单和证据索引作为补充 Sheet/HTML 区域，不改变原十列顺序。

报告门禁校验 Excel、HTML、JSON 的用例 ID、状态、证据数量、哈希和统计完全一致。不一致时整批报告不能交付。

## 20. 完整工作流

1. 识别原生增强模式或兼容模式；
2. 读取 `report.json` 或 Excel；
3. 非标准 Excel 生成映射并等待确认；
4. 执行 E0–E4 准备度检查；
5. 建议但不强制搭配现有七个 Skill；
6. 收集实际目标、凭据引用、数据和可选只读数据库信息；
7. 侦察 Web 页面和接口上下文，不执行未授权业务动作；
8. 生成 `execution-profile.json` 和 `run-manifest.json`；
9. 校验用例来源、白名单动作、断言和清理；
10. 展示整批执行预览和风险；
11. 用户确认后生成 `approval.json` 并锁定哈希；
12. Runner 校验版本、哈希、目标、凭据引用和规则库；
13. 执行 Fixture、Web/API、只读数据库断言和清理；
14. 追加写入原始事件并采集证据；
15. 按断言来源和状态规则生成 `run-result.json`；
16. 同源回填 Excel/HTML；
17. 执行报告一致性门禁；
18. 返回双格式报告、证据目录和人工处理清单。

## 21. Skill 文件设计

`SKILL.md` 保持在 500 行内，只包含：

- 触发边界和独立定位；
- 最高优先级准确性铁律；
- E0–E4 准备度门禁；
- 原生/兼容输入路由；
- 执行预览和确认；
- Runner 调用契约；
- 状态和最终交付门禁；
- 精简自检清单。

按需 references 包含：

- 输入映射与准备度；
- 风险、凭据和数据生命周期；
- Web 定位与修复；
- 断言、知识规则和多层冲突；
- CI、证据和报告协议；
- Runner 命令和适配器说明。

执行 Schema、规则 JSON 和模板作为 assets 或 Runner 包资源，不把大段机器协议加载进模型上下文。

## 22. 第一版范围与非目标

第一版包含：

- Web/API 混合业务链路；
- `report.json` 与 Excel 输入；
- 非标准 Excel 映射确认；
- Playwright Web/API 核心；
- 可选只读数据库；
- 执行准备度、风险预览和批量确认；
- 定位修复提案；
- 技术规则和少量高风险经验；
- 交互与受控 CI；
- 证据归档和 Excel/HTML 回填。

第一版不包含：

- 移动端自动化；
- 性能压测；
- 数据库写入；
- 远程云执行服务；
- 完整行业业务知识包；
- 自动向 Jira、禅道等平台提交缺陷；
- 无确认的定位自修复；
- HTML/CSV/Markdown/纯文本直接执行；
- 任意 Shell、任意 SQL 或任意 AI 生成代码执行。

## 23. 测试策略

### 23.1 Skill 行为测试

按 `writing-skills` 执行 RED-GREEN-REFACTOR。先运行没有 Skill 的基线压力场景，记录模型是否：

- 缺地址/账号仍启动执行；
- 静默映射非标准 Excel；
- 把执行器异常判为研发 Bug；
- 根据域名猜测环境；
- 在日志中泄露 Token；
- 定位失败后乱点或静默自修复；
- 业务断言失败后重试并覆盖证据；
- 修改原预期使当前实现通过；
- 用行业经验替代项目规则；
- Excel/HTML 生成不一致结果。

Skill 完成后用同一批场景验证所有行为门禁，并补充新的合理化漏洞测试。

### 23.2 Runner 测试

- Schema 与协议合同测试；
- 同输入/同清单的确定性测试；
- Web/API/数据库适配器单元测试；
- 演示站点的端到端混合链路；
- 哈希篡改、目标变更、过期授权拒绝测试；
- SQL 只读和多语句阻断测试；
- 凭据与证据脱敏测试；
- 定位失败和修复提案测试；
- 业务失败不重试、基础设施异常仅重试一次；
- 清理失败和人工清理清单测试；
- 知识规则适用条件、例外和版本测试；
- Excel/HTML/JSON 一致性与视觉渲染测试；
- Windows/Linux 交互与 CI 兼容测试。

## 24. 验收标准

发布前必须满足：

1. Skill 可被 Codex、Claude Code 和 CC Switch 发现；
2. Skill 单独安装即可理解工作流，不依赖另外七个包；
3. Runner 使用锁定版本，协议和规则库版本可追踪；
4. 标准 `report.json`、标准 Excel、非标准 Excel 均通过输入合同；
5. 所有非标准 Excel 映射都经过用户确认；
6. E0–E2 不启动正式执行，E3 等待确认，E4 才允许 Runner；
7. 清单确认后任何修改都会使授权失效；
8. Web/API/只读数据库混合样例执行成功；
9. 定位失败不乱点，修复需确认；
10. 业务断言失败不重试，基础设施异常最多重试一次；
11. 数据库写入和多语句被阻断；
12. 用户凭据和敏感数据不进入清单、日志和报告；
13. 业务结果与运行状态严格分离；
14. 多层必须断言冲突判为不通过，未定义冲突进入待定；
15. 行业经验判定带来源、版本、置信度和自动判定标识；
16. 测试数据可追踪，清理失败生成清单；
17. 第一次失败、重试和证据不可覆盖；
18. Excel、HTML 和 JSON 状态、证据和统计一致；
19. 交互模式和受控 CI 模式均通过安全门禁；
20. Skill 行为测试、Runner 自动化测试、报告视觉验证和 `skill-review` 全部通过。

## 25. 发布策略

- 新 Skill 作为 `skills/web-api-test-execution-evidence/` 独立发布；
- Runner 作为 `@saitamasans/testing-runner` 独立版本化；
- Skill 声明支持的 Runner 协议与版本范围；
- 协议不兼容升级必须提高主版本；
- 技术规则和行业经验规则随 Runner 发布并记录版本；
- 初次发布先提供本地交互样例和受控 CI 样例，再开放更多适配器；
- 发布说明明确列出自动执行边界、数据风险、依赖和不支持范围。
