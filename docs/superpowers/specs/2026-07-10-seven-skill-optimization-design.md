# 七个测试 Skill 原位优化设计

## 1. 目标

在保留七个 Skill 的文件名、数量、独立职责和原有表达习惯的前提下，修复上一轮审计发现的触发描述、执行门禁、工作流可跟踪性、规则冲突和 Token 冗余问题。

本次采用“文件内结构化优化”：只修改现有七个 Markdown 文件的内部内容，不合并 Skill，不删除完整版或精炼版，不把正文拆到 `references/`、`scripts/` 或其他 Skill。

## 2. 不可变约束

以下内容不得改变：

1. 七个 Skill 文件一个不少，文件名保持不变。
2. 完整版与精炼版同时保留；完整版承担完整规则覆盖，精炼版服务上下文较小或指令理解能力较弱的模型。
3. 保留中文测试工程师语气、编号层级、表格优先的表达方式。
4. 保留 P0/P1/P2、资料分级、强核对门禁、待确认、合理假设、置信度、聊天版与表格版等既有术语。
5. 保留现有业务风险、测试维度、场景模板和高风险专项，不因压缩而删除有效测试思想。
6. 不将七个 Skill 包装为新的目录型 Skill，不新增替代入口，不改变用户现有复制和调用习惯。

## 3. 允许修改的范围

每个文件仅允许进行以下改动：

1. 在文件顶部增加仅含 `name` 与 `description` 的 YAML frontmatter；`name` 使用稳定英文标识，正文标题和文件名保持原样。
2. 在核心原则前后补充一条明确、可验证的“最高优先级铁律”。
3. 增加 `- [ ]` 格式的执行进度清单，并对不可跳过步骤使用 `⚠️ REQUIRED`、对阻塞条件使用 `⛔ BLOCKING`。
4. 将正文已有的触发语纳入 `description`，正文中的触发场景表可保留，以兼容不读取 frontmatter 的模型。
5. 删除同一文件内部完全重复或近义重复的说明，但保留用户原有句式中的核心表达。
6. 修复跨 Skill 的字段顺序冲突、输出触发冲突和授权门禁缺口。
7. 补充最终交付前的可执行检查，不改变原来的测试结论口径。

## 4. 统一规则

### 4.1 测试用例十列顺序

七个 Skill 涉及正式测试用例表时，统一使用：

`用例 ID → 所属模块 → 用例标题 → 验证功能点 → 前置条件 → 测试步骤 → 预期结果 → 优先级 → 执行结果（通过 / 不通过 / 未执行） → 备注`

原有字段不删除，只调整出现冲突的顺序。用例 ID、模块分割行、编号、黑体、执行结果默认值等原规则继续保留。

### 4.2 文件生成

- 用户明确要求 Excel、Word、WPS 或表格文件时，直接生成对应文件。
- 用户只要求分析、测试点、用例内容或聊天结果时，先交付聊天内容，不擅自创建文件。
- 生成文件后必须重新检查 Sheet、字段顺序、可编辑性和内容完整性。

### 4.3 正式服安全门禁

正式服 Skill 在任何写库、发奖、扣资产、修改配置、越权验证或可能影响真实用户的操作前，必须获得明确授权；缺少授权时只输出只读验证方案和待确认问题。

### 4.4 事实与推断

所有 Skill 继续遵守：事实、接口契约、测试经验风险、合理假设、待确认规则和无依据猜测必须分层；无依据内容不得写成确定预期。

## 5. 七个文件的针对性优化

### 5.1 单接口用例生成与对齐_完整版Skill_v0.3.md

- 保留完整业务覆盖和详细模板。
- 合并文件内部重复出现的双产出、错误码固定表达、最终输出和禁止事项说明。
- 补充可跟踪执行清单和阻塞门禁。
- 不将它改写成精炼版，也不删除面向复杂模型的详细解释。

### 5.2 精炼版_单接口用例与对齐skill_v0.3.md

- 保持短句、直接规则和线性执行顺序。
- 不增加长篇原理解释；新增内容优先使用清单和短表。
- 与完整版保持相同结论，但不追求相同细节量。

### 5.3 多接口链路测试用例生成skill_v0.6_精炼执行版.md

- 保留 M1–M7 输入模式、链路准入、置信度、可观测性和数据清理规则。
- 压缩样式、配色和重复输出说明，避免“精炼执行版”正文过重。
- 明确禁止接口全排列和无依据链路推断的阻塞位置。

### 5.4 根据需求-用例生成_skill.md

- 保留测试工作台和多种输出模式定位，不拆分成多个 Skill。
- 强化模式选择清单，避免每次都机械执行全量模式。
- 保留“六个 Skill/完整版”这一用户触发习惯。

### 5.5 正式服验证-用例生成 Skill.md

- 保留 L0–L3 分层、不建议正式服执行清单和执行前后检查。
- 增加正式服写操作 `⛔ BLOCKING` 授权门禁。
- 收紧越权、并发、配置和真实资产操作的执行条件。

### 5.6 测试用例-审计与评_Skill_V1.md

- 保留完整审计、标准审计、降级审计和不进入审计的分级。
- 增加“默认只审计，不自动重写”的执行门禁。
- 将问题清单、漏测依据和冗余合并检查转换为可跟踪步骤。

### 5.7 测试视角-需求澄清 Skill.md

- 保留“只做需求澄清、不生成测试用例”的硬边界。
- 修正缺少一级标题和元数据的问题。
- 保留具体产品提问示例，压缩重复模板，强化产品回答后的复审门禁。

## 6. 表达习惯保护方法

实施时使用以下规则避免“改得不像原作者”：

1. 优先移动、合并和删重，不优先重写。
2. 必须改写时沿用原文件的词汇、句长、编号和表格结构。
3. 不将中文规则改成英文正文；英文仅用于 frontmatter 的稳定 `name`。
4. 不引入营销化、抽象化或软件架构式表达。
5. 不删除原有典型业务词，如金额、余额、库存、奖励、状态、权限、幂等、并发、数据一致性。
6. 修改前后抽样对比每个文件的标题、核心术语和关键句式，确认语气保持一致。

## 7. 验证方案

### 7.1 静态验证

对七个文件逐一验证：

- 文件名和文件数量未变化。
- frontmatter 可解析且只含 `name`、`description`。
- 每个文件存在执行清单、最高优先级约束、阻塞门禁和最终自检。
- 无 TODO、FIXME、TBD 或占位符残留。
- 涉及测试用例表的文件使用统一十列顺序。
- 正式服 Skill 存在明确授权门禁。

### 7.2 风格回归验证

对比修改前后的：

- 中文字符占比与标题风格；
- 核心术语保留率；
- 表格数量和关键业务示例；
- 完整版与精炼版的长度差异；
- 七个 Skill 的独立触发边界。

### 7.3 行为验证

使用代表性场景分别验证：

- 一句话需求不会直接生成正式用例；
- 单接口与多接口场景不会混用；
- 只有用例、没有需求时只做降级审计；
- 正式服未授权写操作被阻断；
- 用户未要求文件时不擅自创建 Excel；
- 完整版和精炼版产生一致的核心结论，但解释深度不同。

## 8. 完成标准

只有同时满足以下条件才视为完成：

1. 七个原始文件名、数量和职责保持不变。
2. 用户原有表达习惯和核心测试思想得到保留。
3. 上一轮审计指出的元数据、清单、门禁、字段冲突和冗余问题得到修复。
4. 静态验证、风格回归和代表性行为验证全部通过。
5. 未自动推送、未创建 PR、未修改远端仓库。
