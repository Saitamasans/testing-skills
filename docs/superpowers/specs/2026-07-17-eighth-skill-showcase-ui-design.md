# 第八个 Skill 演示项目 UI 重设计

日期：2026-07-17
状态：设计已确认，等待书面规格复核
适用范围：`web-api-test-execution-evidence`、`@saitamasans/testing-runner`、SkillMart 本地隔离演示系统

## 1. 背景

当前第八个 Skill 已能真实执行 Web/API 测试、生成证据并回填报告，但交互可视化仍是注入到页面右上角的工具型浮层。观众能看到执行，却不容易理解为什么开始、当前验证什么、断言如何得出、何时结束以及最终交付了哪些材料。SkillMart 被测页面本身也只有最小功能，缺少完整业务产品应有的信息架构、状态和视觉层次。

本次改造不改变测试数据、接口契约、执行判定和证据规则。目标是把现有能力包装成可理解、可参观、可录制、可复现的完整演示产品，同时保证 Runner 驾驶舱可用于任意 Web/API 项目，不与 SkillMart 绑定。

## 2. 目标与非目标

### 2.1 目标

- 让第一次接触第八个 Skill 的普通测试人员在首屏理解本次执行的输入、范围、目标和输出。
- 让测试负责人能查看执行进度、断言依据、异常分类、缺陷根因和证据完整性。
- 把 SkillMart 重做为无需 Runner 也能独立理解和操作的电商演示项目。
- 在 Runner 执行 Web、API 或混合测试时持续提供清晰视觉反馈。
- 保持测试用例（Test Case）和测试用例（Test Cases）的中英文双名称约定。
- 保持未执行、通过、不通过、待定四种业务状态及现有七种运行状态。
- 保证正式 PNG 不含驾驶舱，桌面录屏保留完整驾驶舱和操作提示。

### 2.2 非目标

- 不新增或生成测试用例（Test Cases）。
- 不修改标准十列顺序、Excel/HTML/JSON 的判定来源或报告一致性门禁。
- 不让演示 UI 参与定位、点击、断言或业务结果判定。
- 不通过域名、数据库或数据特征猜测环境性质。
- 不把 SkillMart 专属内容写进通用 Runner 的状态模型。
- 不把演示项目扩展成真实支付、电商后台或需要外部账号的生产系统。

## 3. 受众与设计原则

首要受众是第一次接触第八个 Skill 的普通测试人员，次要受众是测试负责人和管理者。信息顺序先回答“现在在做什么”，再提供技术细节。

采用以下设计原则：

1. 先解释，再执行。浏览器打开后先展示 3 至 5 秒执行准备页，然后自动开始，不增加第二次人工确认。
2. 被测系统是主角。Web 执行时真实页面占据主要画面，Runner 只负责解释，不替代页面。
3. 视觉反馈跟随动作。每次点击、输入、请求和断言都显示对应的人类可读说明。
4. 结果有来源。预期、实际、断言来源和证据文件之间可以追溯。
5. 异常不混淆。业务不通过、待定、执行器错误、基础设施错误和人工处理保持明确区分。
6. 演示不影响准确性。视觉增强不能改变页面布局、点击目标、响应数据和最终状态。
7. 信息饱满但不拖慢。只有开场和用例预告使用短暂停留，真实动作不增加无意义等待。

## 4. 整体产品旅程

浏览器顶部使用稳定的五阶段导航，当前阶段高亮，已完成阶段保留完成标记。

### 4.1 执行准备

浏览器打开后展示 3 至 5 秒，随后自动开始。页面包含：

- Skill 名称与 Runner 锁定版本；
- 用例来源文件和测试用例（Test Cases）总数；
- 目标 origin、执行模式和风险等级；
- Web/API/混合类型与动作数量；
- 将要生成的 Excel、HTML、JSON、截图、日志和 Trace；
- 凭据已脱敏、审批已确认、数据清理已配置等准备状态。

准备页只展示既有审批结果，不替代 Codex 中的执行确认，也不提供再次点击“开始执行”的按钮。

### 4.2 用例预告

每条测试用例（Test Case）执行前展示短暂预告，包含序号、ID、中文标题、所属模块、验证目标、前置条件、预计动作数和证据类型。预告停留时间应短于 1.5 秒，并允许 `prefers-reduced-motion` 环境取消动画。

### 4.3 实时执行

Web 场景采用“真实被测页面 + 右侧执行驾驶舱”。左侧显示真实点击、输入、提交、跳转和业务反馈；右侧持续展示：

- 当前测试用例（Test Case）和总体进度；
- 当前动作及其人类可读说明；
- 验证目标和前置条件；
- 预期结果、实际结果、断言来源和判定；
- 通过、不通过、待定、未执行数量；
- manifest hash、目标 origin 和已用时间。

Web 动作执行前使用不拦截鼠标的短暂高亮标出目标控件，并显示“准备点击创建订单”“正在填写优惠券”等说明。高亮、驾驶舱和转场均不参与定位与断言。

API 动作把左侧主画面切换为 API 流水视图，展示方法、路径、脱敏请求头、请求体、响应状态、响应体、耗时、变量提取和逐条断言。混合用例在真实网页和 API 流水视图之间自动切换，避免 API 执行期间出现空白页。

### 4.4 证据收集

执行结束后显示收尾步骤：业务清理、测试数据复位、截图保存、请求响应落盘、日志完成、Trace 关闭、Excel/HTML 回填和报告一致性校验。每个步骤有独立状态和失败原因。

如果证据保存或报告一致性门禁失败，界面必须明确标记交付未完成，不能用业务用例已执行代替完整交付。

### 4.5 结果中心

结果中心展示：

- 四状态汇总与运行状态；
- 可筛选的测试用例（Test Cases）列表；
- 当前测试用例（Test Case）的步骤、预期、实际、断言和证据；
- 按根因聚合的缺陷列表及受影响用例；
- 待定项的冲突来源和人工审核说明；
- Excel、HTML、run-result.json、截图、日志、API 证据和 Trace 的明确入口；
- 报告一致性门禁结果和最终交付结论。

## 5. SkillMart 演示系统

SkillMart 是独立可用的本地电商演示系统，不依赖外部数据库、支付平台或账号。默认首屏直接进入可操作的商品中心，不增加营销落地页。

### 5.1 信息架构

- **商品中心**：真实本地商品图片、商品名称、分类、价格、库存、状态、搜索和分类筛选。
- **结算流程**：商品选择、数量调整、优惠券输入、优惠明细、应付金额和创建订单。
- **订单中心**：订单编号、商品明细、金额、订单状态、状态时间线、订单查询和取消操作。
- **运行数据**：当前库存、已创建订单、数据更新时间和本地演示数据状态。

页面使用可预测的顶部导航，不使用装饰性营销 Hero。品牌和商品必须在首个视口内明确出现，商品中心是第一屏主要工作区。

### 5.2 完整状态

必须覆盖正常、加载中、空数据、库存不足、优惠券无效、订单创建失败、订单不存在、无权访问、订单不可取消和数据重置等状态。错误使用页面内反馈，不使用 `window.alert()`。

### 5.3 自动化兼容

- 保留现有接口路径、请求方法、字段语义和预置缺陷。
- 保留或兼容现有 `data-testid`，新增交互控件使用稳定且语义明确的 `data-testid`。
- 动态文案、图标和视觉层级不得成为唯一定位方式。
- 固定格式组件使用稳定尺寸，加载和状态变化不能造成无关布局跳动。
- 使用本地静态图片资源，不让网络图片失败影响演示和测试。

## 6. Runner 驾驶舱架构

### 6.1 通用状态模型

扩展现有 `VisualProgressState`，但不改变 `RunObserver` 对业务执行的单向观察关系。驾驶舱只消费以下事件：

- run started/completed；
- case started/completed；
- action started/completed；
- evidence and cleanup progress；
- report projection and consistency result。

业务判定仍由执行器和 `run-result.json` 产生，UI 不得回写判定。

### 6.2 Web 展示

驾驶舱继续封装在隔离的 Shadow DOM 中，使用固定层、稳定尺寸和 `pointer-events: none`，不拦截被测页面操作。默认位于右侧；执行前根据当前目标控件的可见边界选择左侧或右侧，避免遮挡。目标边界不可用时保持默认右侧。

SkillMart 的桌面布局预留右侧安全区域，因此正式演示中驾驶舱不需要频繁移动。其他项目不依赖该安全区域，使用通用的自动侧边选择。

页面刷新或跳转会移除注入节点，下一次 Runner 事件必须重新注入完整驾驶舱。重复注入只保留一个 host。

### 6.3 API 展示

API-only 执行使用完整的 API 流水页，不再使用空白页面加小面板。请求、响应和断言区域使用稳定高度和可滚动内容，长 JSON 折叠展示；默认脱敏 Authorization、Cookie、token、密码和连接信息。

### 6.4 正式证据与录屏

- 桌面教程录屏保留准备页、驾驶舱、控件高亮、API 流水和结果中心。
- 正式 Web PNG 截图前临时隐藏驾驶舱和控件高亮，截图完成后恢复。
- API 证据保存真实脱敏请求、响应、状态码、耗时和断言，并生成可视化流水页供人工查看。
- Trace 保留原始执行过程；报告中的证据路径必须指向真实存在的文件。

## 7. 视觉系统

采用已确认的“专业质量中心”方向：

- SkillMart 使用中性浅灰和白色工作界面，正文使用深石墨色；
- Runner 驾驶舱使用偏黑的深绿色中性色，与被测页面形成稳定对比；
- 主强调色使用克制的质量绿，红色只表示不通过，灰色表示待定和非活动状态；
- 不使用紫蓝 AI 渐变、装饰光球、过度圆角、卡片套卡片或营销式大标题；
- 数据数字使用等宽或 tabular figures；
- 控件、表格、状态行和画面在 1920×1080 录制中保持清晰；
- 保持桌面优先，同时确保窄视口不出现文字重叠和不可访问控件。

状态视觉规则：

- **通过**：保持正常行底色，使用小型成功标记，不铺大面积绿色。
- **不通过**：使用淡红行背景、红色状态文字和明确失败原因。
- **待定**：使用淡灰行背景，展示口径冲突来源，明确其不是研发缺陷。
- **未执行**：使用中性背景和次级文字，不与待定混淆。

## 8. 错误处理与安全

- 被测页面错误、断言不通过、待定、Runner 错误、基础设施错误和人工处理分别显示，不合并成单一失败。
- 凭据和敏感字段在 UI、日志、HTML、JSON、截图和 Trace 辅助元数据中统一脱敏。
- 驾驶舱注入失败不得影响业务执行；记录视觉层错误并继续保存执行证据。
- 结果中心找不到产物时显示缺失文件及生成阶段，不创建虚假入口。
- 执行被阻塞时保留准备信息、阻塞原因和可操作的补充材料清单。

## 9. 测试策略

实现遵循 RED、GREEN、REFACTOR：先新增失败测试，再修改生产代码。

### 9.1 单元和契约测试

- 五阶段状态转换与事件映射；
- Web/API/混合视图切换；
- 当前动作、预期、实际、断言来源和四状态渲染；
- 驾驶舱左右侧选择、重新注入和单 host 约束；
- 敏感信息脱敏；
- 正式截图隐藏视觉层并在截图后恢复；
- 结果中心产物路径和缺失状态；
- 现有报告一致性与业务状态回归。

### 9.2 真实执行测试

- 从标准十列输入生成 manifest、审批并执行完整 SkillMart Web/API 混合用例；
- 验证控件被真实点击、接口被真实调用、数据被真实清理；
- 验证通过、不通过、待定、未执行均可展示且判定一致；
- 验证正式 PNG 不含驾驶舱，录屏画面包含驾驶舱；
- 验证 Excel、HTML、JSON、日志、截图和 Trace 均存在且互相一致。

### 9.3 视觉验收

- 使用 1920×1080 桌面视口和至少一个窄视口截图；
- 检查首屏信息、长标题、长路径、长 JSON、错误文本和四状态无重叠；
- 检查真实点击目标没有被驾驶舱遮挡；
- 检查准备页、执行中、API 流水、证据收集和结果中心均有非空画面；
- 重新录制第八个 Skill 的完整未剪辑教程，观众能看见从确认执行到结果交付的连续过程。

## 10. 验收标准

1. 浏览器打开后 5 秒内说明执行来源、范围、目标和输出，并自动开始。
2. 任意时刻可以明确看出当前测试用例（Test Case）、当前动作、预期、实际和断言状态。
3. SkillMart 不启动 Runner 时仍是可理解、可操作、视觉完整的电商演示项目。
4. Web、API 和混合用例均有对应的非空实时画面。
5. 四种业务状态及其颜色、语义与报告保持一致。
6. UI 不参与定位、点击和判定，现有执行准确性与报告一致性测试保持通过。
7. 正式证据 PNG 不含驾驶舱，桌面录屏包含完整驾驶舱。
8. 结果中心能打开真实存在的 Excel、HTML、JSON、截图、日志和 Trace。
9. 1920×1080 录屏中页面、驾驶舱和关键文字无重叠、遮挡或截断。
10. 完整自测后交付新验收导航、逐阶段截图和未剪辑录屏，不以口头结论代替证据。
