# 第八个 Skill 可视化执行与 1080p60 教程设计

日期：2026-07-16
状态：已确认设计，待实施

## 1. 目标

增强 `web-api-test-execution-evidence` 及其配套 Runner，使测试人员在可见执行时可以一眼确认：当前正在执行哪条测试用例（Test Case）、执行到哪个动作、Web/API 实际做了什么、断言结果是什么，以及整体进度如何。

本次只增强第八个 Skill，不改变前七个 Skill 的名称、职责、互斥路由或正式十列测试用例（Test Cases）格式。

## 2. 用户体验原则

- 视觉反馈优先。执行开始后必须立即出现可见反馈，不能让测试人员依赖终端日志或执行结束后的报告判断是否正在运行。
- 信息一目了然。默认展示当前测试用例（Test Case）、当前动作、状态和总进度，不要求用户理解 Runner 内部结构。
- 可见执行默认开启进度面板；只有显式传入 `--progress off` 时关闭。
- CI 和无头模式不显示执行面板，不增加浏览器依赖或等待人工操作。

## 3. 采用方案

采用“Web 页面浮层＋API 专用执行看板”的混合方案。

### 3.1 Web 与混合 Web/API 场景

- Chromium 使用可见交互模式启动并自动最大化，默认占满当前桌面可用区域。
- 在被测页面右上角显示固定执行浮层，不改变业务页面布局。
- 浮层使用隔离容器、`pointer-events: none` 和不参与可访问名称/定位器匹配的结构，避免影响点击、填充、选择和断言。
- 页面导航后自动恢复浮层；不能因为 `web.goto` 或页面刷新而丢失进度信息。

### 3.2 API-only 场景

- 可见交互模式下仍启动浏览器，显示独立的全屏 API 执行看板。
- 看板实时展示请求方法、目标路径、响应状态码、断言、结果和执行耗时。
- CI、`--browser headless` 或 `--progress off` 时，API-only 执行不为可视化额外启动浏览器。

## 4. 执行面板信息

默认显示：

- 主 Skill：`web-api-test-execution-evidence`；
- manifest hash 的短摘要和目标 origin；
- 当前测试用例（Test Case）序号与总数；
- 用例 ID、中文标题和所属模块；
- 当前动作序号与总动作数；
- 动作类型，例如 `web.click`、`api.request`、`api.assert`；
- Web 定位目标，或 API 方法与路径；
- 当前状态：准备、执行中、通过、不通过、待定、未执行；
- 已通过、不通过、待定、未执行的实时统计；
- 总体进度条和已用时间。

不在面板中展示密码、token、数据库连接串、请求密钥或其他敏感值。请求头和请求体继续遵循现有脱敏规则。

## 5. 状态与数据流

Runner 在执行过程中产生以下可视化事件：

1. `run.started`：显示 manifest、目标、测试用例（Test Cases）数量和动作数量；
2. `case.started`：切换当前测试用例（Test Case）；
3. `action.started`：显示即将执行的 Web/API/数据库动作；
4. `action.passed|failed|pending|blocked`：更新动作结果和实时统计；
5. `case.completed`：固定显示该用例最终四状态；
6. `run.completed`：切换到汇总视图并保留窗口，等待测试人员查看或关闭。

现有 `run-result.json` 仍是唯一业务判定来源。可视化面板只消费执行事件，不自行判断通过、不通过、待定或未执行。

## 6. 正式证据与可视化隔离

- 视频录制中保留执行面板，帮助观众理解正在执行的测试用例（Test Case）。
- Runner 保存正式 Web 证据 PNG 前临时隐藏面板，截图完成后立即恢复，确保业务证据不被遮挡。
- Playwright Trace 保留真实执行过程；面板事件不替代请求、响应、断言或 Trace 证据。
- Excel、HTML 和 `run-result.json` 的四状态规则保持不变：未执行、通过、不通过、待定。

## 7. 命令与兼容性

- `interactive + browser auto/visible`：默认全屏并显示进度面板；
- `--progress off`：关闭面板，保留原有执行逻辑；
- `ci`：强制无头且无面板；
- `--browser headless`：无面板；
- 原有命令未增加必填参数，旧调用可以继续运行。

第八个 Skill 的固定执行说明需要补充：交互执行默认提供全屏可视化进度；用户无需额外配置；不需要可视化时可显式关闭。

## 8. 错误处理

- 面板初始化失败不能静默降级：交互模式应输出明确错误，并阻止把“无可视化执行”冒充为教程结果。
- 页面导航导致浮层丢失时自动恢复；连续恢复失败则记录 `visual_progress_error`。
- 面板更新失败不能改变业务测试结果，但必须写入运行日志和最终说明。
- 浏览器无法最大化时允许退化为当前可用最大窗口，但必须在日志中记录实际窗口尺寸。

## 9. 测试策略

按 TDD 实施：

1. 先增加失败测试，证明当前 API-only 交互执行不会启动可视化浏览器、可见浏览器不会最大化、Runner 不会上报当前用例和动作；
2. 实现最小可用的可视化事件模型与浏览器面板；
3. 验证 Web、API-only、混合场景以及 `--progress off`、CI、headless 分支；
4. 验证正式证据 PNG 不包含面板；
5. 使用 SkillMart 真实测试用例（Test Cases）执行一次完整前向测试。

必须覆盖：窗口最大化、默认开启、关闭开关、用例标题、动作进度、API 请求/响应、四状态统计、导航恢复、结束汇总停留和敏感信息脱敏。

## 10. 重新录屏与交付

第八个 Skill 专用教程重新录制，其他七个 Skill 只作为测试用例（Test Cases）来源简要说明，不占用主体时长。

录制要求：

- 1920×1080；
- 60 FPS；
- 鼠标指针可见；
- 浏览器全屏；
- 执行面板默认开启；
- 从 Skill 调用、执行预览、确认开始，连续展示 Web 点击、API 请求/断言、证据生成、状态回填和最终汇总；
- 保留完整未剪辑版，并生成 20 分钟以内的教程版；
- 使用 FFprobe 和全片解码验证分辨率、帧率、时长及文件完整性。

## 11. 验收标准

- 测试人员无需查看终端即可确认当前执行的是哪条测试用例（Test Case）；
- 浏览器在录屏中以全屏或当前桌面最大可用尺寸展示；
- API-only 执行有实时可视化反馈；
- 面板默认开启，`--progress off`、CI 和 headless 行为符合设计；
- 正式证据 PNG 不被执行面板遮挡；
- 四状态、Excel/HTML/JSON 一致性和缺陷根因聚合逻辑不发生回归；
- 新录屏为 1920×1080、60 FPS，包含真实连续执行过程，而不是结果页轮播。
