# 增量 Skill：正式服验证用例生成 Skill

## 最高优先级铁律

**未获得正式服写入授权、测试账号、执行时间窗和风险联系人前，只能输出只读验证方案。**

⛔ **BLOCKING：写库、发奖、扣资产、修改配置、越权验证、并发请求或任何可能影响真实用户的操作，必须先得到明确授权。**

这是原子化硬门禁，不得用“老板要求”“赶时间”“仅一次”“低价值”“灰度环境”或“先执行后补审批”绕过。只有四项全部满足，才允许设计相应的最小写操作；任何一项缺失都必须明确输出“写操作范围阻塞，未执行任何生产写入或高风险操作”，并且只输出正式服只读验证用例。普通 L0 页面 GET、公开入口访问、合法只读配置/监控查询在具备对应只读权限时可安排执行，不要求写操作授权或内部写入账号，但不得访问未授权数据。不得在未授权输出中提供可直接执行的写入、配置、安全测试或并发步骤、命令和参数。

### BLOCKING 准入清单 ⚠️ REQUIRED

以下项目是主动写入、产生副作用、安全探测或并发请求的准入门禁，必须逐项得到明确事实；不得自行补全、默认已同意或用业务负责人催促代替。纯 L0 只读验证只要求目标与范围明确、公开资源可访问或具备对应合法只读权限，不要求补齐写操作专用的授权、内部写入账号、时间窗和风险联系人：

- [ ] **目标**：已明确本次要证明的上线结果和停止条件。
- [ ] **范围**：已明确正式服/灰度范围、版本、功能、对象及请求/数据上限。
- [ ] **授权 ⚠️ REQUIRED**：已取得书面明确授权，逐项写明允许的写入、发奖/扣资产、配置、安全验证或并发动作；沉默和笼统上线许可不算授权。
- [ ] **账号 ⚠️ REQUIRED**：已指定带“正式服验证”标记的内部测试账号及操作者；不得改用真实用户账号或真实用户数据。
- [ ] **时间窗 ⚠️ REQUIRED**：已指定开始/结束时间、灰度或维护窗口。
- [ ] **联系人 ⚠️ REQUIRED**：已指定风险联系人、监控值守人、升级路径和立即停止条件。
- [ ] **数据**：已限定可识别、低价值、可追踪的数据及资产上限；禁止触碰真实用户资产。
- [ ] **监控**：日志、traceId、指标、告警和只读核对权限已就绪。
- [ ] **最小执行**：动作、次数、并发度和影响面已收敛到授权范围内的最小值；默认单账号、单请求、单条数据。
- [ ] **收尾与归档**：已明确核对、留痕、数据标记、归档和另行审批的清理方式。

**判定规则：** 授权、账号、时间窗、联系人任一项未勾选，或高风险动作未被授权逐项点名，立即停止生产写操作设计；只给只读替代方案。该阻塞只约束写操作范围，不把具备合法访问条件的 L0 只读验证判为阻塞。全部勾选也不代表高风险动作自动放行，仍须按 L0–L3 和专项限制选择最小风险方案。

## 一、Skill 定位

本 Skill 用于已经上线到正式服、预发布环境、生产环境、灰度环境后的功能验证。

它不替代测试服全量测试用例，也不用于覆盖所有边界、异常、并发、性能和破坏性场景。

它的目标是生成一套：

1. 能证明核心功能已在正式服可用的验证用例
2. 能覆盖主流程闭环的最小充分执行集
3. 尽量少写库、少造数、少产生脏数据
4. 不影响真实用户、不改变线上关键配置
5. 不执行高风险破坏性测试
6. 可回溯、可归档、可向产品/研发/运营证明上线状态的正式服验证资产

## 三、核心原则

正式服验证不是测试服全量回归。

必须遵循以下原则：

| 原则      | 说明                            |
| ------- | ----------------------------- |
| 主流程优先   | 优先验证入口、核心操作、核心结果、记录/后台/日志闭环   |
| 少量写操作   | 能只读就只读；四项硬门禁通过且动作被明确授权后，才使用最小可控数据 |
| 低价值数据   | 奖励、支付、礼物、积分、抽奖仅限指定内部账号和获批上限；不使用真实用户资产 |
| 可识别数据   | 测试账号、测试房间、测试昵称、测试备注必须可识别      |
| 可回滚/可归档 | 产生的数据要可查询、可解释、可清理或可归档         |
| 不影响用户   | 不改全局配置、不影响真实用户路径、不制造异常流量，不把真实用户当测试对象 |
| 不做破坏性测试 | 并发、刷接口、风控、批量注册、恶意越权不在正式服执行；灰度受控验证也须专项授权 |
| 分层验证    | 页面、接口、数据、记录、监控按风险选择，不机械全测     |
| 风险留痕    | 不适合正式服执行的用例要单独列出，不要直接删除       |
| 验证可诊断   | 每条用例要能判断失败原因，不写“检查是否正常”这类空泛描述 |

拿到需求或测试服用例后，先判断正式服验证类型：

| 类型          | 特征                   | 输出策略                 |
| ----------- | -------------------- | -------------------- |
| P0 主链路正式服验证 | 核心功能必须在线上跑通          | 门禁未通过时只生成只读用例；通过后才生成获批的最小写入用例 |
| P1 低风险补充验证  | 不一定阻断上线，但能提高信心       | 选择性生成                |
| P2 展示/兼容验证  | 低风险、只读、无写库           | 可轻量纳入                |
| 禁止正式服执行     | 会污染数据、影响用户、改变配置、触发风控 | 不生成执行用例，放入“不建议正式服执行” |
| 条件执行        | 需要运营/研发/客服/后台支持      | 四项硬门禁通过且动作被逐项授权后，标注最小执行条件和风险 |

## 五、正式服验证分层模型

### 1. L0 只读验证

不产生业务数据，仅查看页面、入口、文案、配置、记录。

| 场景          | 是否推荐正式服执行 |
| ----------- | --------- |
| 活动入口是否展示    | 推荐        |
| Banner 是否跳转 | 推荐        |
| 活动时间、规则说明展示 | 推荐        |
| 用户记录查询      | 推荐        |
| 后台记录查询      | 推荐        |
| 配置只读检查      | 推荐        |

### 2. L1 低风险写操作

会产生少量数据，但可控、可识别、影响小。L1 仍是生产写操作：只有明确授权、指定内部账号、指定时间窗和风险联系人全部就绪后才能生成执行用例，否则降级为 L0 只读验证。

| 场景           | 是否推荐正式服执行   |
| ------------ | ----------- |
| 内部测试账号绑定邀请码  | 已授权、指定内部账号与时间窗内只执行 1 次 |
| 测试房间进入/发言/上麦 | 已授权、指定内部账号与时间窗内按获批次数执行 |
| 低价值礼物直刷      | 已授权且明确资产上限后，用指定内部账号执行 1 次 |
| 抽奖 1 次       | 已授权且明确扣点/发奖上限后，用指定内部账号执行 1 次 |
| 生成一条可识别记录    | 已授权、指定内部账号与时间窗内生成 1 条并归档 |

### 3. L2 受控高风险写操作

可能涉及奖励、资产、库存、后台配置。通用上线授权不足以放行 L2；必须取得对应动作的专项书面授权，并指定内部账号、时间窗、风险联系人、资产上限、监控和回滚/停止方案。

| 场景       | 是否推荐正式服执行      |
| -------- | -------------- |
| 抽奖发奖     | 仅限指定内部账号、低价值资产和获批时间窗内 1 次 |
| 修改后台活动配置 | 不作为常规测试执行；仅限获批灰度配置，由指定操作者双人复核并具备回滚方案 |
| 客服补偿     | 仅限获批演练和指定内部账号，不向真实用户发放测试补偿 |
| 贡献值大额触发  | 不在正式服构造；改用隔离环境或只读核对自然结果 |

### 4. L3 禁止/不建议正式服执行

| 场景          | 原因               |
| ----------- | ---------------- |
| 批量注册新用户     | 污染用户数据           |
| 高频抽奖/批量抽奖   | 污染奖池和奖励记录        |
| 并发压测        | 影响真实用户           |
| 恶意刷风控       | 可能触发真实处罚         |
| 修改活动开始/结束时间 | 影响全量用户           |
| 关闭正式服活动开关   | 影响真实用户           |
| 构造百万贡献值     | 资损和脏数据风险高        |
| 非授权越权攻击     | 安全风险，应在测试/预发环境执行 |
| 大量接口重放      | 可能触发限流、风控、误告警    |
| 删除/回滚线上数据   | 高风险；测试人员不得自行执行，须独立变更审批和指定操作者 |

## 六、正式服验证用例设计流程

生成正式服验证用例时，必须按以下步骤处理：

### 第一步：确认正式服验证目标

先执行 BLOCKING 准入清单，不以提问后继续为默认路径：

| 核对项 | 必须取得的明确事实 | 缺失时处理 |
| --- | --- | --- |
| 目标与范围 | 功能、版本、全量/灰度范围、指定对象、次数或数据上限 | 只读核对已发布事实，不写入 |
| 正式服写入授权 | 书面逐项点名允许的写库、发奖/扣资产、配置、安全或并发动作 | 阻塞所有生产写操作和相应执行步骤 |
| 测试账号 | 指定内部测试账号、操作者、账号标记和允许资产上限 | 不得用真实用户账号替代，只输出只读方案 |
| 执行时间窗 | 明确开始/结束时间、维护或灰度窗口 | 不执行主动写入、产生副作用、安全探测或并发请求；普通 L0 只读访问不受此写操作门禁限制 |
| 风险联系人 | 值守人、失败联系人、升级路径和停止条件 | 不执行主动写入、产生副作用、安全探测或并发请求；普通 L0 只读访问不受此写操作门禁限制 |
| 数据与资产 | 可识别数据、低价值资产、单次上限及归档方式 | 不发奖、不扣资产、不造数 |
| 监控与收尾 | 日志、traceId、告警、核对、留痕、归档和清理审批 | 不生成对应写操作用例 |

任一硬门禁缺失时，到此停止写操作设计，首段明确写明“写操作范围阻塞”，后续只输出 L0 只读验证策略和十列只读用例。若公开页面可访问或已具备对应合法只读权限，L0 可安排执行并独立判断通过/不通过/待观察；不得因缺少写操作门禁把纯 L0 整体判为阻塞。不得把写操作包装成“条件执行用例”继续输出。

### 第二步：从测试服全量用例中筛选正式服用例

优先保留：

1. 入口可见性
2. 页面可访问性
3. 核心主流程
4. 最小数据写入闭环
5. 奖励或状态结果验证
6. 记录查询
7. 后台/日志/监控核对
8. 关键失败态的轻量验证

默认剔除：

1. 大量边界组合
2. 批量异常输入
3. 并发压测
4. 高价值奖励验证
5. 后台全局配置修改
6. 恶意越权攻击
7. 风控刷量
8. 破坏性数据构造

### 第三步：确定正式服测试数据策略

| 数据类型 | 推荐策略                     |
| ---- | ------------------------ |
| 测试账号 | 使用内部账号，昵称或备注包含“正式服验证”    |
| 邀请关系 | 仅准备 1 组邀请人 + 被邀请人        |
| 房间   | 使用内部测试房间或低流量房间           |
| 礼物   | 使用低价值礼物，避免高价值资产          |
| 抽奖   | 四项门禁和抽奖专项授权齐备后，指定内部账号只抽 1 次；否则只读核对自然流量记录 |
| 奖励   | 四项门禁和资产上限齐备后验证低价值奖励；否则只读核对后台记录 |
| 后台配置 | 只读核对，不改正式配置              |
| 日志   | 使用测试账号 ID、traceId、时间范围查询 |
| 数据清理 | 默认标记并归档；清理/回滚须另行审批，由指定操作者执行，不自行删除 |

### 第四步：生成正式服验证用例

正式服用例必须满足：

1. 每条用例步骤少
2. 每条用例目标明确
3. 不重复验证低价值点
4. 写库动作最小化
5. 每个写库动作都有记录核对
6. 每个风险动作都有备注说明
7. 每个失败态都尽量采用不产生副作用的方式验证
8. 对不能在线上执行的测试点，放入“不建议正式服执行”清单

## 七、正式服验证用例表字段

正式服测试用例表统一使用以下十列，且这是全局唯一顺序。所有正式用例表、示例表、模板表和导出文件都必须逐列一致，不得调换、缺列或新增列：

| 用例 ID | 所属模块 | 用例标题 | 验证功能点 | 前置条件 | 测试步骤 | 预期结果 | 优先级 | 执行结果（通过 / 不通过 / 未执行） | 备注 |
| ----- | ---- | ---- | ----- | ---- | ---- | ---- | --- | -------------------- | -- |

字段要求：

1. 用例 ID 使用模块缩写-三位序号，例如 PROD-001、BIND-001、DRAW-001。
2. 每个模块前插入【模块分割行】。
3. “验证功能点”必须用 1、2、3 编号列出该用例实际验证什么。
4. “前置条件”“测试步骤”“预期结果”必须编号。
5. “执行结果（通过 / 不通过 / 未执行）”默认填写“未执行”。
6. 如果生成 Excel / Word / WPS，字体设置为黑体。
7. 不新增“测试数据”“是否冒烟”“是否回归”“是否自动化”等列。
8. 测试数据要写入前置条件、测试步骤或备注中。
9. “验证功能点”固定为第 4 列，“备注”固定为第 10 列；模块分割行也必须占满同一十列。
10. 未通过 BLOCKING 门禁时，表中只能出现 L0 只读用例，不得出现写入、安全测试、配置变更或并发用例，即使执行结果填写“未执行”也不允许。

## 八、正式服验证模块建议

根据功能复杂度选择模块，不要求机械全用。

| 模块           | 适用场景                 |

| 正式服记录与后台核对   | 前台记录、后台记录、日志、报表      |
| 正式服前后端限制轻量验证 | 状态拦截、接口拒绝、无副作用失败态    |
| 正式服监控与告警观察   | 错误日志、接口成功率、异常告警      |
| 正式服收尾与数据归档   | 测试数据标记、清理、归档说明       |

## 九、正式服验证优先级规则

| 优先级 | 判断标准                             |
| --- | -------------------------------- |
| P0  | 正式服必须验证的核心入口、核心主流程、核心数据结果、资损相关闭环 |
| P1  | 重要但可条件执行的记录、后台、日志、轻量异常、兼容验证      |
| P2  | 低风险展示、文案、弱网、低频兼容、非阻断验证           |

正式服 P0 不等于测试服 P0。
测试服 P0 里的并发、越权、批量边界、风控刷量，不一定适合正式服执行。

## 十、正式服验证中的前后端限制策略

正式服仍要验证前后端一致性，但不能使用高破坏方式。

| 场景 | 正式服策略 | 强制准入条件 |
| --- | --- | --- |
| 活动未开始 | 只读查看状态；不强行修改正式时间 | 只读权限即可；主动灰度请求仍须指定内部账号和窗口 |
| 活动已结束 | 自然结束后只读验证，或在测试活动验证 | 不修改正式时间 |
| 点数不足 | 优先用已处于不足状态的指定内部账号只读/单次验证 | 明确授权、指定账号、指定时间窗、风险联系人 |
| 未登录/token 过期 | 可做无业务写入的失败态验证 | 不得携带真实用户 token；主动请求须在指定时间窗 |
| 单次非法参数请求 | 仅允许一个无副作用、无敏感数据的参数失败态，不做 fuzz | 明确接口安全授权、指定内部账号、指定时间窗、风险联系人和 1 次上限 |
| 受控单例越权验证 | 正式服默认不执行；确有必要时只允许两个指定内部账号之间访问一条自有测试记录，不枚举、不扫描、不触碰真实用户数据 | 安全负责人专项书面授权、账号 A/B、指定时间窗、风险联系人和审计日志全部齐备 |
| 轻量连点 | 只允许指定内部账号对获批操作做一次双击验证，不扩大为重放或并发 | 明确授权、指定内部账号、指定时间窗、风险联系人及最多 2 个请求 |
| 并发 | 正式服全量环境不执行；仅可在隔离灰度范围按明确并发上限验证 | 研发与运维专项授权、指定内部账号、指定灰度窗口、风险联系人、限流和停止条件 |
| 库存仅剩 1 | 不在正式服主动构造，避免影响真实奖池 | 改用隔离环境；正式服只读观察自然结果 |
| 活动开关关闭 | 正式服只读核对；不把配置变更当测试步骤 | 只有独立变更审批、指定操作者、灰度范围、双人复核、指定窗口和回滚方案齐备时才可按变更单执行 |

上述任何强制准入条件缺失时，对应场景不得进入用例表；只能给出只读核对方式。一般“允许上线验证”不等价于越权、并发、配置变更或真实资产操作的专项授权。

## 十三、执行前检查清单

以下清单是执行门禁，不是提醒。主动写入、产生副作用、安全探测或并发请求只要有一个对应必需项未确认，就必须阻塞该操作范围并退回只读方案。普通 L0 页面 GET、公开入口、合法只读配置/监控查询不要求写操作授权、内部写入账号、专项写操作时间窗或写操作风险联系人，但必须具备对应合法访问权限，且不得访问未授权数据。

| 检查项            | 是否必须 |
| -------------- | ---- |
| 确认验证目标、版本、功能及全量/灰度/指定对象范围 | 是 |
| 确认正式服版本已发布     | 是    |
| 确认活动/功能开关状态    | 是    |
| 取得逐项点名允许动作、次数和资产上限的书面明确授权 | 写入/副作用/安全探测/并发时是；纯 L0 否 |
| 确认测试账号可登录      | 需要账号访问时是；公开 L0 否 |
| 确认测试账号不是真实用户   | 主动业务操作时是；公开或聚合只读 L0 否 |
| 确认是否允许写库       | 涉及写库时是；纯 L0 否 |
| 确认是否允许发奖/扣点/支付 | 涉及资产时是；纯 L0 否 |
| 确认测试房间/测试对象    | 涉及对应业务对象时是；纯 L0 否 |
| 确认测试数据可识别、低价值、可追踪且不触碰真实用户资产 | 是 |
| 确认后台、日志、traceId、监控和告警查询权限 | 是 |
| 确认失败时联系人       | 写入/副作用/安全探测/并发时是；纯 L0 否 |
| 确认执行时间窗口       | 写入/副作用/安全探测/并发时是；纯 L0 否 |
| 确认最小执行次数、影响面、停止条件和升级路径 | 主动业务或高风险请求时是；纯 L0 否 |

## 十六、正式服验证结果判断

| 结果    | 判断标准                            |
| ----- | ------------------------------- |
| 通过    | P0 用例全部通过，P1 无阻断问题，监控无异常        |
| 有风险通过 | P0 通过，但存在 P1/P2 问题或监控轻微异常       |
| 阻塞    | 任一 P0 主流程失败、资损异常、数据不一致、接口严重报错   |
| 回滚建议  | 主流程不可用、发奖/扣款/资产异常、影响真实用户、告警持续扩大 |
| 待观察   | 功能可用但存在异步延迟、日志异常、少量用户反馈待确认      |

授权、内部写入账号、执行时间窗或风险联系人任一缺失时，写操作范围必须判为“阻塞”，不得用“有风险通过”降级放行。纯 L0 只读验证在公开资源可访问或具备合法只读权限时独立判断“通过 / 不通过 / 待观察”；只有用户目标必须依赖被阻塞的写操作且只读证据无法完成目标时，整体结论才是“阻塞”。

## 十七、正式服验证最终自检 ⚠️ REQUIRED

输出正式服验证用例前，必须检查：

1. 是否只保留正式服适合执行的最小充分用例。
2. 是否覆盖入口、主流程、结果、记录/后台、监控闭环。
3. 是否避免批量造数、大量写库、高频接口、并发压测。
4. 是否明确哪些测试不建议正式服执行。
5. 是否每条写库用例都有数据归档或清理说明。
6. 是否每条用例都有“验证功能点”。
7. 是否每条用例前置条件、步骤、预期结果都有编号。
8. 是否执行结果默认是“未执行”。
9. 是否说明测试账号、测试数据、测试时间窗口。
10. 是否能在失败时快速定位是前端、接口、数据、配置、权限还是环境问题。
11. 如果导出为文件，测试用例表是否使用黑体。
12. 是否在任何写操作前同时取得明确授权、指定内部账号、指定时间窗和风险联系人；任一缺失时是否只输出只读方案。
13. 是否所有正式用例表严格使用全局十列顺序，且“验证功能点”为第 4 列、“备注”为第 10 列。
14. 是否把单次非法参数、受控单例越权、轻量连点、并发、配置变更分别绑定到专项授权和最小上限，而没有用通用授权替代。
15. 是否禁止使用真实用户账号、真实用户数据或真实用户资产完成验证。

## 十二、互斥路由与组合调用

正式服、线上、生产环境、上线后验证或生产冒烟由本 Skill 作为主 Skill，安全门禁始终优先。若目标是线上单接口或多接口链路，可建议对应接口 Skill 作为最多一个辅助 Skill；辅助 Skill 只能补充接口测试设计，不能降低或绕过本 Skill 的生产四项门禁。

调用前必须先展示主 Skill、辅助 Skill、职责分工和“最终只生成一套结果”，并暂停等待确认。用户确认前，不加载辅助 Skill、不执行接口专项分析、不生成最终文件；拒绝后继续本 Skill，并说明当前结果不包含相应接口专项深度。聊天、Excel 总览和 HTML 总览均记录调用与确认结果。

## 十三、Excel 与 HTML 双交付合同

聊天请求不创建文件。用户明确要求文件时，同一份报告 JSON 同时生成 `.xlsx` 和单文件离线 `.html`；任何格式都不能绕过生产门禁。正式用例十列顺序不变，执行结果固定为 `未执行 / 通过 / 不通过 / 待定`。`待定`表示已执行但需求、产品、研发或测试口径存在歧义，暂时不能定性为缺陷，备注由测试人员自行决定。

模块分割行不设下拉；Excel 中 `不通过`整行淡红、`待定`整行淡灰，其他状态保留风险等级、模块和优先级原色。HTML 含搜索、模块/优先级/状态筛选、冻结表头、四状态统计、下拉、按报告隔离的 `localStorage` 本地自动保存，不得发起外部网络请求。

```bash
node scripts/render-test-assets.mjs --input report.json --output-dir output --basename 正式服验证用例
```

必须验证两个文件均存在且可打开后再返回两个链接；未通过安全门禁时，两个载体都只能包含获准的只读方案。
