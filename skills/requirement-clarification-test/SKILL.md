# 测试视角需求澄清 Skill（开测前需求补全分析）

### 1. Skill 定位

本 Skill 是独立的“开测前需求补全分析”能力，只用于从测试视角分析产品需求是否清晰、完整、可测、可验收，并输出需要产品回答的核对问题。

本 Skill 不生成测试用例，不生成测试点，不生成需求追踪矩阵，不生成接口用例，不生成完整测试资产。

核心目标：

1. 找出需求中不明确、不可测、边界不清、异常缺失、权限不清、前后端规则不一致、数据一致性不明的问题。
2. 把测试视角的疑问整理成产品能直接回答的问题表。
3. 支持测试与产品多轮澄清：产品回答后继续复审，直到需求达到可写用例、可执行、可验收的程度。
4. 在需求未达到可开测标准前，不进入正式测试用例生成阶段。

#### 1.1 最高优先级边界铁律

> **本 Skill 只做需求澄清，不生成测试用例。只要仍有 P0 未确认，就必须标记 `⛔ BLOCKING`，不得进入测试用例生成。**

`P0 ⛔ BLOCKING` 是固定字面量，必须逐字使用；不得把 `⛔` 替换为 `⚠`，也不得简写成“P0 阻断”或仅写“BLOCKING”。

1. 用户同时说“先澄清”和“顺手写用例”、以工期或上级催促施压，或要求不要停下来时，仍只交付澄清结果。
2. 产品回答后必须先复审；回答了部分 P0 不等于解除阻塞。只要仍有核心规则、主流程、核心预期或数据结果未确认，就继续保持 `⛔ BLOCKING`，不得生成测试用例、测试点表或用例骨架。
3. 只有复审确认所有 P0 已关闭，才可给出最终准入结论；本 Skill 仍不生成用例，只能把后续工作交给独立的测试用例生成 Skill。

### 2. 使用场景

当用户提供以下内容时，优先使用本 Skill：

1. PRD、用户故事、原型说明、验收标准。
2. 老板口头需求、一句话需求、不完整需求。
3. 需求变更说明。
4. 产品补充说明。
5. 测试想判断“需求是否能开测、能不能写用例、还缺哪些边界”。

---

### 3. 硬性边界

1. 本 Skill 只做需求澄清，不写测试用例。
2. 不输出正式测试点表。
3. 不输出需求追踪矩阵。
4. 不输出接口测试用例。
5. 不把合理假设当作已确认规则。
6. 不替产品编造业务规则、边界值、提示文案、权限规则、状态流转。
7. 如果存在 P0 阻断问题，统一标记为 `P0 ⛔ BLOCKING`，只输出澄清结果和产品核对问题，不进入用例生成。
8. 产品回答后，只做复审、补充追问和准入判断；确认可开测后，再交给独立的测试用例生成 Skill。

### 4. 开测准入结论

每次分析需求后，先给出准入结论：

| 结论    | 含义                      | 后续处理               |
| ----- | ----------------------- | ------------------ |
| 可开测   | 核心流程、规则、边界、权限、异常、验收基本明确 | 可进入测试设计或用例生成 Skill |
| 有条件开测 | 主流程明确，但仍有 P1/P2 风险未确认   | 可继续推进，但需标注风险       |
| 不建议开测 | 存在 P0 缺口，不确认会导致用例基于猜测   | 停止用例生成，先让产品回答问题    |
| 不可开测  | 目标矛盾、流程断裂、预期不可验证        | 退回产品重新打磨需求         |

判断原则：

只有“不确认就无法判断核心预期”的问题才算 P0 阻断。
不是所有需求不完整都阻断；P1/P2 可以作为风险记录继续澄清。

### 5. 需求澄清维度

分析需求时，重点检查以下维度：

| 维度     | 要检查的问题                         |
| ------ | ------------------------------ |
| 业务目标   | 需求解决什么问题？成功标准是什么？              |
| 使用角色   | 谁能看？谁能操作？不同角色差异是什么？            |
| 入口路径   | 从哪个页面、按钮、弹窗、接口、任务进入？           |
| 操作对象   | 操作的是订单、用户、活动、奖品、配置、库存还是其他对象？   |
| 主流程    | 从开始到完成是否闭环？每一步系统如何响应？          |
| 前置条件   | 是否需要登录、权限、特定状态、配置开关、测试数据？      |
| 成功结果   | 成功后页面、接口、数据库、状态、消息、日志如何变化？     |
| 失败结果   | 失败时是否提示、落库、扣减、回滚、允许重试？         |
| 边界值    | 金额、数量、次数、时间、库存、长度、分页、状态边界是否明确？ |
| 等价类    | 有效值、无效值、空值、枚举值、异常值是否明确？        |
| 权限归属   | 是否区分自己的数据、他人数据、下级数据、跨组织数据？     |
| 状态流转   | 初始状态、中间状态、终态、非法状态是否明确？         |
| 异常恢复   | 刷新、返回、取消、断网、超时、接口失败后怎么处理？      |
| 幂等并发   | 连点、重复提交、多端同时操作、库存仅剩 1 时怎么处理？   |
| 前后端一致性 | 前端限制后端是否也校验？绕过前端直接调接口是否拦截？     |
| 数据一致性  | 列表、详情、统计、导出、消息、数据库是否同步？        |
| 历史兼容   | 老数据、旧版本、缺字段数据是否受影响？            |
| 验收标准   | 是否能判断通过/失败，而不是“正常展示”“符合预期”？    |
| 本轮范围   | 本轮明确包含什么？不包含什么？哪些后续再做？         |

### 6. P0 / P1 / P2 判定规则

| 优先级 | 含义                            | 是否阻断          |
| --- | ----------------------------- | ------------- |
| P0  | 不确认就无法判断主流程、核心规则、核心预期或数据结果    | 阻断            |
| P1  | 影响异常、边界、权限、数据一致性、幂等、并发、前后端一致性 | 通常不阻断，但必须提示风险 |
| P2  | 影响文案、样式、低风险展示、兼容性、体验细节        | 不阻断           |

P0 常见类型：

1. 目标不清：不知道需求要解决什么问题。
2. 角色不清：不知道谁能看、谁能操作。
3. 入口不清：不知道从哪里触发。
4. 对象不清：不知道操作什么数据或资源。
5. 主流程不闭环：不知道操作后系统应该变成什么状态。
6. 核心规则缺失：金额、库存、次数、时间、状态、权限、命中规则不清。
7. 数据结果不清：页面、接口、数据库、下游如何变化不明确。
8. 验收不可测：只有“优化体验”“正常展示”“符合预期”等空泛描述。
9. 需求互相冲突：前后描述矛盾，无法判断正确口径。
10. 前端影响业务结果，但后端是否校验未说明。

### 7. 输出结构

使用本 Skill 时，默认按 7.1～7.5 输出；收到产品回答后再增加 7.6 复审结论。7.0 仅用于内部执行跟踪。

#### 7.0 内部可跟踪执行清单

按顺序在内部检查进度；每一项都要能回指需求原文、产品回答、原型或接口文档等来源。未完成项不得默认为已确认。默认用户输出不得展示复选框或 `CL-01`～`CL-07` 编号；只有用户明确要求查看分析过程或执行清单时才展示。

- [ ] CL-01 准入结论 ⚠️ REQUIRED：先给出当前开测准入、是否可进入用例生成、最大风险和 P0 状态。
- [ ] CL-02 当前理解：整理业务目标、角色、入口、对象、主流程、成功/失败结果和本轮范围，不扩写未知规则。
- [ ] CL-03 已明确内容：用 `C001` 记录已确认规则、来源和限制，避免重复追问。
- [ ] CL-04 具体问题：用 `Q001` 输出产品能直接回答的问题；每个未确认 P0 标记 `P0 ⛔ BLOCKING` 并说明阻断原因。
- [ ] CL-05 本轮不纳入：用 `N001` 固定本轮排除项、原因和后续建议，防止范围发散。
- [ ] CL-06 产品回答复审：逐项记录已回答、未回答、含糊、冲突及新问题；部分回答不得自动关闭整组 P0。
- [ ] CL-07 最终准入 ⚠️ REQUIRED：重新统计未关闭 P0 并给出最终准入。仍有任一 P0 时保持 `⛔ BLOCKING`，不进入用例生成。

#### 7.1 当前开测准入结论

| 项目         | 结论                         |
| ---------- | -------------------------- |
| 开测准入结论     | 可开测 / 有条件开测 / 不建议开测 / 不可开测 |
| 是否可进入用例生成  | 是 / 否                      |
| 当前最大风险     | 用一句话说明                     |
| 是否存在 P0 阻断 | 是 / 否                      |

#### 7.2 当前需求理解

用测试视角把需求整理成可测描述，不扩写未确认规则。

| 项目   | 当前理解     |
| ---- | -------- |
| 业务目标 | 需求想达成什么  |
| 使用角色 | 谁使用      |
| 入口路径 | 从哪里进入    |
| 操作对象 | 操作什么     |
| 主流程  | 大致流程     |
| 成功结果 | 成功后如何变化  |
| 失败结果 | 目前是否说明   |
| 本轮范围 | 当前能确认的范围 |

#### 7.3 已明确内容

用于沉淀产品已经说清楚的规则，避免多轮沟通时重复追问。

| 编号   | 已明确内容     | 来源                      | 备注      |
| ---- | --------- | ----------------------- | ------- |
| C001 | 已确认的规则或范围 | 需求原文 / 产品补充 / 原型 / 接口文档 | 说明限制或风险 |

#### 7.4 产品核对问题表

问题要尽量一次性问全，避免零散追问。
每个问题必须具体、可回答，不能只写“请确认规则”。

| 问题编号 | 优先级      | 核对问题 | 为什么要问  | 建议产品回答格式     | 是否阻断开测 |
| ---- | -------- | ---- | ------ | ------------ | ------ |
| Q001 | P0 ⛔ BLOCKING / P1 / P2 | 具体问题 | 对测试的影响 | 给出可选项或要求明确规则 | 是/否    |

问题写法要求：

1. 问题必须能让产品直接回答。
2. 优先问边界、异常、权限、状态、前后端一致性、数据一致性。
3. 同一业务域的问题尽量一次性归并，不要反复零散追问。
4. 对复杂场景，按“正常、异常、边界、权限、并发、失败恢复、提示对象、提示位置、提示内容”一次性拆问。
5. 不问发散问题，不追问与本轮目标弱相关的内容。
6. 未确认的 P0 问题必须写成 `P0 ⛔ BLOCKING`，并说明为什么阻断开测。

#### 7.5 本轮不纳入

用于防止需求无限发散，明确哪些内容本轮不测、不问、不扩展。

| 编号   | 不纳入内容      | 原因                      | 后续建议         |
| ---- | ---------- | ----------------------- | ------------ |
| N001 | 暂不纳入的能力或场景 | 超出当前需求 / 依赖其他模块 / 产品未规划 | 后续单独提需求或补充说明 |

#### 7.6 复审结论

当产品回答问题后，必须复审：

| 复审项           | 结论                   |
| ------------- | -------------------- |
| 本轮新增已确认内容     | 列出产品已回答且可落规则的内容      |
| 仍未确认的问题       | 列出未回答、回答含糊、回答冲突的问题   |
| 新发现的问题        | 产品回答后引出的新边界、新异常、新冲突  |
| 未关闭 P0 数量与状态   | 逐项列出；任一未关闭即为 `⛔ BLOCKING` |
| 准入结论变化        | 是否从不可开测变为有条件开测 / 可开测 |
| 是否可交给用例 Skill | 是 / 否                |

### 8. 产品核对问题生成原则

当需求缺少细节时，不要只泛泛输出“权限待确认、边界待确认”。
必须转成产品能回答的具体问题。

错误示例：

| 错误问题     |
| -------- |
| 请确认抽奖规则。 |
| 请确认异常情况。 |
| 请确认权限。   |
| 请确认边界值。  |

正确示例：

| 正确问题                                    |
| --------------------------------------- |
| 用户余额不足时，点击抽奖是按钮置灰、弹窗提示，还是允许点击后接口返回失败？   |
| 奖品库存为 0 时，前端是否展示该奖品？用户抽中库存为 0 的奖品时如何处理？ |
| 抽奖扣减顺序是先扣余额再抽奖，还是先判断奖品后扣余额？发奖失败时余额是否回退？ |
| 用户重复点击抽奖按钮时，后端是否保证只扣一次钱、只产生一次抽奖记录？      |
| 普通用户是否只能查看自己的抽奖记录？运营是否能查看全部用户记录？        |

### 9. 常见业务问题模板

#### 9.1 通用需求核对

通用需求直接按第 5 节“需求澄清维度”核对，不再重复列同一组问题。以下 9.2～9.4 仅在命中对应业务域时作为补充，不能替代第 5 节的通用核对。

#### 9.2 抽奖 / 发奖类需求核对

当需求涉及抽奖、奖品、补偿池、库存、发奖、概率、次数、余额、活动时，必须重点追问：

| 类型    | 应追问的问题                               |
| ----- | ------------------------------------ |
| 抽奖资格  | 谁能抽？游客、未登录、黑名单、封禁用户能否抽？              |
| 消耗规则  | 抽奖消耗什么？余额、钻石、次数、券、积分？何时扣减？           |
| 余额不足  | 用户没钱、余额不足时，按钮是否置灰？是否弹窗？接口返回什么？       |
| 次数限制  | 单用户、单设备、单 IP、单活动、每日是否限制次数？           |
| 活动时间  | 活动未开始、进行中、已结束时分别如何展示和拦截？             |
| 奖品库存  | 库存为 0、库存不足、库存仅剩 1 时如何处理？             |
| 中奖逻辑  | 抽中哪个奖品由前端还是后端决定？概率在哪里配置？             |
| 发奖失败  | 抽中后发奖失败，是否补发、回滚、重试、记录异常？             |
| 扣减顺序  | 先扣钱、先扣次数、先扣库存、先生成记录的顺序是什么？           |
| 并发    | 多人同时抽中最后一个奖品时，谁成功？失败方提示什么？           |
| 重复点击  | 用户连点或重复请求时，是否只扣一次、只生成一条记录？           |
| 提示对象  | 提示给当前用户、运营后台、客服、日志系统中的谁？             |
| 提示位置  | 页面 toast、弹窗、按钮下方、抽奖结果页、站内信？          |
| 提示内容  | 余额不足、库存不足、活动结束、网络失败分别提示什么？           |
| 记录查询  | 抽奖记录、中奖记录、消耗记录、发奖记录在哪里可查？            |
| 数据一致性 | 用户余额、奖品库存、中奖记录、发奖状态、后台统计是否一致？        |
| 后端校验  | 绕过前端直接请求抽奖接口，后端是否校验余额、次数、库存、活动状态、权限？ |

#### 9.3 导出 / 报表类需求核对

| 类型   | 应追问的问题                 |
| ---- | ---------------------- |
| 导出范围 | 导出当前筛选结果、当前页，还是全部数据？   |
| 字段口径 | 导出哪些字段？字段顺序、名称、格式是什么？  |
| 权限   | 谁能导出？导出是否受数据范围权限限制？    |
| 数据量  | 最大导出多少条？超出后同步失败还是异步任务？ |
| 空数据  | 无数据时导出空文件还是提示无数据？      |
| 一致性  | 导出数据是否与列表、详情、统计一致？     |
| 失败处理 | 导出失败是否可重试？是否生成失败任务记录？  |

#### 9.4 支付 / 订单 / 库存类需求核对

| 类型    | 应追问的问题                    |
| ----- | ------------------------- |
| 下单资格  | 谁能下单？是否需要登录、实名、风控通过？      |
| 金额规则  | 价格、优惠、运费、实付金额如何计算？        |
| 库存规则  | 何时锁库存、扣库存、释放库存？           |
| 支付失败  | 支付失败、取消、超时、重复支付如何处理？      |
| 退款规则  | 谁能退款？哪些状态能退？金额如何计算？       |
| 并发    | 库存仅剩 1 时，多人下单谁成功？         |
| 幂等    | 重复提交订单是否生成重复订单？           |
| 状态流转  | 待支付、已支付、已取消、已退款等状态如何迁移？   |
| 数据一致性 | 订单、支付单、库存、余额、消息、后台统计是否一致？ |

---

### 10. 产品回答后的复审规则

产品补充回答后，不直接进入用例生成，必须先复审：

1. 将产品明确回答的内容写入“已明确内容”。
2. 检查是否仍有 P0 未确认；未关闭项逐一标记 `P0 ⛔ BLOCKING`。
3. 检查产品回答是否与原需求、原型、接口、历史规则冲突。
4. 检查回答是否引出了新的边界、异常、权限或数据一致性问题。
5. 对含糊回答继续追问，例如“按正常逻辑处理”“和之前一样”“应该不允许”。
6. 重新给出准入结论；即使产品已回答部分 P0，只要核心规则仍未确认，结论仍为不建议开测或不可开测。
7. 只有所有 P0 已关闭，并达到“可开测”或用户明确接受不涉及 P0 的“有条件开测风险”后，才允许交给测试用例生成 Skill。

---

### 11. 达到可写用例的标准

需求达到以下标准，才算可进入后续测试用例生成：

| 标准      | 判断                     |
| ------- | ---------------------- |
| 能判断预期   | 知道正常、异常、边界、权限、失败时应该怎样  |
| 能准备数据   | 知道账号、权限、状态、配置、测试数据如何准备 |
| 能执行步骤   | 知道入口、操作对象、操作路径         |
| 能判断结果   | 知道页面、接口、数据库、状态、下游如何验证  |
| 能控制范围   | 知道本轮包含什么、不包含什么         |
| 无 P0 阻断 | 核心规则、主流程、权限、边界、验收无关键缺失 |

输出结论：

| 判断        | 后续              |
| --------- | --------------- |
| 已达到用例生成准入 | 可交给测试用例生成 Skill |
| 仅达到有条件开测  | 可继续，但必须保留风险说明   |
| 未达到准入     | 继续产品核对，不生成正式用例  |

---

### 12. 最终自检 ⚠️ REQUIRED

输出前检查：

1. 是否明确本 Skill 只做需求澄清，没有生成测试用例。
2. 是否给出开测准入结论。
3. 是否输出已明确内容。
4. 是否输出产品核对问题表。
5. 是否区分 P0/P1/P2。
6. P0 是否真正阻断核心预期，而不是把所有细节都打成 P0。
7. 产品核对问题是否具体、可回答。
8. 边界、异常、权限、状态、前后端一致性、数据一致性是否尽量一次性问全。
9. 是否输出本轮不纳入。
10. 是否说明产品回答后需要复审。
11. 是否避免编造产品没写的规则。
12. 是否在 P0 未确认时停止进入用例生成。

## 13. 互斥路由与组合调用

用户明确要求只澄清需求、判断能否开测、暂时不要测试点或测试用例时，本 Skill 作为主 Skill。用户要正式测试设计或用例时改走需求工作台；已有用例评审改走用例审计；接口或正式服目标改走对应主 Skill。

本 Skill 通常不需要辅助 Skill。确有组合需要时最多一个辅助 Skill，并先展示主 Skill、辅助 Skill、职责分工和“最终只生成一套结果”，暂停等待用户确认。用户确认前，不加载或执行辅助 Skill；拒绝后继续澄清并说明覆盖限制。

本 Skill 不生成正式测试用例，因此不使用执行状态下拉，也不套用正式用例的 Excel 与 HTML 默认双交付。澄清结束后如用户要求进入用例生成，应重新路由到需求工作台，并保留已经确认的事实、未决问题和证据边界。
