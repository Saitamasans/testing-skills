# 增强版-灰盒测试用例生成 Skill

版本：v1.0.0

以需求测试工作台为完整地基，以需求为测试真值，以相关前端、后端、Admin、接口文档、数据库、缓存、异步链路、代码 Diff 和历史测试资料为增强证据，识别真实业务分支、关联影响和回归范围，最终生成一套供测试人员手工执行的高质量正式测试用例。

## 最高优先级铁律

1. **需求决定系统应该怎样，代码说明系统当前怎样实现。**需求真值、实现证据和运行事实必须分开。
2. 实现资料完整不等于需求完整。L0 或存在未关闭 P0 时，不得因代码完整而生成正式测试用例。
3. 代码、接口文档、数据库、Admin 和历史实现不能替产品确认业务目标、核心权限、数值范围、状态流转、生效时间、成功/失败结果、数据结果和验收标准。
4. 需求与代码冲突时，按已确认需求设计预期，把代码记录为实现差异并让差异进入测试点；不得静默采用代码规则。
5. 全仓搜索后只深读需求相关链路，不机械读取所有代码，也不以覆盖所有 `if/else` 为目标。
6. 开发前旧代码用于影响分析；开发后 Diff 用于实现增强。不得把旧代码当作新需求实现。
7. 代码疑点只能形成增强测试点、回归点或待动态验证线索；未经执行不得认定为Bug，不得写“已复现”或“已确认 Bug”。
8. 最终只生成一套灰盒增强的正式测试用例，不另建第二套黑盒用例。
9. 用例追求最小充分、可执行、可诊断、可追踪，不靠数量体现完整。
10. 不进行真实接口调用，不连接数据库、Redis 或消息系统，不执行代码审计。
11. **代码分析不是独立交付目标。**每个高价值实现发现必须进入正式测试用例、待产品确认或待动态验证之一；不得只输出代码分析结论而不转化为测试设计。
12. **正式测试用例是第一核心交付物。**默认输出中它必须是最完整、最重要的部分，代码证据只用于提高用例准确度、细节、可执行性和回归判断。

## 定位与边界

本 Skill 是完整、独立、自包含的测试用例生成 Skill。它继承需求测试工作台的输入质量判断、L0/L1/L2/L3、P0 门禁、范围控制、场景法、等价类、边界值、决策表、状态迁移、权限与数据归属、幂等和并发、数据一致性、前后端一致性、隐含分支、测试点提取、去冗余、风险优先级、用例独立性和文件交付规范，并增加真实实现证据链分析。

它本质上是面向手工测试人员的测试用例生成 Skill，不是静态漏洞审计、接口/Web 自动执行、Bug 自动确认、全仓分支覆盖或用代码替代产品需求的工具。不以发现漏洞、列举类和方法或覆盖全部 `if/else` 为主要交付；不执行真实接口、数据库、Redis、MQ 或 WebSocket。最终目标始终是结合已确认需求和真实实现证据，生成一套更准确、更详细、更可执行的正式测试用例。

它的增强价值是：让角色、入口、参数、边界、权限、对象归属、状态和数据流更贴近真实系统；识别 Android、iOS、H5、Admin 多端差异；形成页面、接口、数据库、缓存、异步和最终业务结果的闭环；按公共方法、字段、配置和调用关系生成最小充分回归范围；覆盖即时、延迟和最终一致性，以及老数据、旧版本和历史状态兼容。

## 输入准入与互斥路由

最少输入是“一份需求资料 + 至少一种与当前需求相关的实现资料”。

- 需求资料：PRD、用户故事、需求变更、产品确认稿、产品答复、验收标准、原型说明或口头需求记录。
- 实现资料：Android、iOS、H5/Web、Admin、后端代码、接口文档、OpenAPI/Swagger/Apifox、数据库结构、migration、Model、Redis、Queue/MQ、WebSocket、PR、Commit、代码 Diff、开发自测、真实请求响应、已有用例或历史相关 Bug。
- 资料不要求全部齐全。缺少某一层时标记证据缺口，继续现有资料支持的分析；不编造缺失层，不停止其他分析。

以下场景不适用，立即轻量路由：

| 输入/目标 | 处理 |
|---|---|
| 只有需求文档 | 推荐 `requirement-test-workbench`；不扫描代码仓库，不把“没有代码”当成需求阻塞 |
| 只有一句话需求且只需澄清 | 推荐 `requirement-clarification-test` |
| 只有接口文档，没有业务需求 | 推荐 `single-api-test-full` 或 `multi-api-flow-test` |
| 只有代码 | 不编造业务预期；路由到代码审计或接口审计能力，不生成正式需求用例 |
| 只有已有用例且要评审 | 推荐 `test-case-quality-audit` |
| 正式服轻量验证 | 推荐 `production-verification-test` |
| 自动执行已有用例 | 推荐 `web-api-test-execution-evidence` |

一个任务只选择一个主 Skill，最多一个辅助 Skill。需要辅助 Skill 时，先展示主 Skill、辅助 Skill、职责分工和“最终只生成一套结果”；用户确认前不得加载辅助 Skill、执行辅助分析或生成最终文件。

## 双重输入分级

先独立判断需求质量，再判断实现资料成熟度。

| 需求级别 | 判断与处理 |
|---|---|
| L0 | 口头或一句话需求；进入 P0 澄清门禁 |
| L1 | 有目标但缺规则、异常、权限或验收；评审、假设和待确认 |
| L2 | 有流程、字段、规则和部分验收；可做标准设计 |
| L3 | PRD、原型、状态、权限和验收充分；按目标生成正式资产 |

| 实现级别 | 判断与处理 |
|---|---|
| G0 | 只有需求，没有相关实现资料；路由 `requirement-test-workbench` |
| G1 | 需求 + 单一实现资料；执行基础灰盒增强 |
| G2 | 需求 + 多层实现资料；执行标准灰盒增强 |
| G3 | 需求 + 多端代码 + 数据链路 + 本次 Diff + 历史资料或运行证据；执行完整灰盒增强 |

`L0 + G3` 仍然是需求阻断，代码不能绕过 P0。不得因代码完整而生成正式测试用例。

## L0 与 P0 门禁

L0 或仍有未关闭 P0 时，只输出：

- 输入质量判断、当前需求理解和已明确内容；
- 主流程缺口与 P0/P1/P2 确认问题；
- 代码揭示的系统现状，且逐项标记为“当前实现证据”；
- 代码辅助发现的精准澄清问题、本轮不纳入和准入结论。

禁止输出正式测试用例、正式接口用例、正式追踪矩阵和完整测试资产，也不得把当前实现证据包装为产品确认规则。

## 判断开发阶段

### A. 开发前影响分析

当前只有旧版本代码时，把它标记为旧系统基线，只用于定位影响模块、现有流程、开发风险和回归范围。不得把旧代码当成新需求实现，不得据此认定新需求存在实现缺陷。输出初版灰盒用例与开发后复核项。

### B. 开发后实现增强

存在本次 PR、Commit、Diff 或修改后代码时，对比已确认需求与本次实现，分析新增、修改、删除分支，更新测试点和回归范围，识别实现偏差、公共方法影响和旧功能回归，并记录仓库、分支和 Commit。

### C. 阶段无法确认

标记“开发阶段待确认”，不把当前代码认定为最终实现；继续完成已确认需求与现有证据支持的测试设计。

## 固定工作流

1. 做轻量资料分类：依据文件名、路径、扩展名、仓库职责、标题、首段、接口路径和需求关键词。
2. 提取需求关键词与业务对象。
3. 在用户提供或明确指向的仓库中全仓搜索。
4. 做相关性筛选，只保留与需求存在明显关联的资料。
5. 对命中链路执行相关链路深读。
6. 建立需求、实现和运行证据模型。
7. 判断开发阶段，构建差异、影响和回归范围。
8. 选择测试技术，生成候选测试点并去冗余。
9. 生成一套正式用例、少量辅助 Sheet 和 HTML 总览。
10. 完成格式、内容、证据和文件自检，停止无关扩展。

大型接口文档先搜关键词和路径；大型代码仓库先检索再深读。明显无关模块直接排除。

## 需求相关全链路定位

根据实际存在的资料定位：

需求规则 → 页面入口 → Android/iOS/H5 交互 → 客户端参数来源 → 接口文档 → 后端路由 → 中间件 → 鉴权/加密 → Controller → Service → Model → Admin 配置 → 数据库字段与索引 → Redis → Queue/MQ → WebSocket → 下游列表、详情、统计、结算、消息或通知。

每项结论必须尽量回指文件路径、方法或类、行号或明确定位、仓库、分支和 Commit。无法定位时写“未提供”或“未定位”，禁止编造。

对 Android/iOS/H5/Web/Admin，提取页面入口、角色可见性、按钮状态、输入限制、默认值、本地校验、客户端参数来源、空值处理、类型转换、枚举映射、缓存、提示、刷新和多端差异。建立“客户端参数来源 → 请求字段 → 接口定义 → 后端读取 → 校验/持久化/下游”的链；识别前后端差异并让差异进入测试点。

## 真值优先级与需求—实现对照

需求真值优先级：产品确认后的需求 → 产品补充答复 → 明确验收标准 → 已确认业务规则。

当前实现来源：本次代码 Diff → 当前有效前后端代码 → 接口文档 → Admin 配置 → 数据库结构 → Redis/Queue/WebSocket → 历史代码。

运行事实来源：真实请求响应 → 真实页面行为 → 数据库查询结果 → Redis 状态 → Queue 消费证据 → WebSocket 消息 → 自动化执行证据。

内部建立需求—实现对照矩阵，至少记录规则 ID、已确认需求、需求来源、前端/客户端、接口、后端、Admin、数据库、缓存/异步、结论、测试处理和证据位置。结论只允许：一致、部分一致、实现缺失、实现超出需求、需求与实现冲突、多端实现不一致、无法确认、不适用。

默认只展示高价值冲突；用户明确说“完整分析”“按完整版”“全链路分析”或“全量灰盒分析”时才输出完整矩阵。

## 实现层分析

### 前端、客户端、接口和后端

- 前端限制必须检查后端是否重新校验，并覆盖绕过前端直接请求。
- 提取 HTTP 方法、路径、Header、Token、加密、必填、类型、枚举、默认值、错误码、响应、幂等键、路由、中间件、权限、对象归属、状态机、事务、锁、重复提交、异步任务、异常和下游调用。
- 重点识别客户端传了但后端未读取、后端读取但文档未定义、客户端可控而后端未重新校验、前后端类型/枚举不一致、响应与客户端解析不一致、权限和归属风险。

### Admin、数据库和异步链路

- Admin：开关、默认值、上下限、生效范围/时间、配置权限、历史配置、前后台冲突和同步。
- 数据库：类型、长度、可空、默认值、唯一索引、关联、状态枚举、老数据、空数据、脏数据、迁移和事务边界。
- Redis：Key、写入/失效时机、TTL、DB 一致性、权限或状态缓存残留。
- Queue/MQ：生产者、消费者、延迟、重试、重复消费、死信、补偿和幂等。
- WebSocket：事件、订阅范围、推送对象、状态内容、HTTP 一致性、多账号/多端同步和断线恢复。

## 实现发现必须转化为测试设计

代码分析不是独立交付目标。逐项把高价值发现放入以下三类之一：

1. **正式测试用例**：已有确认需求和可观察验证路径；例如前端范围与后端范围不一致时，生成前端边界及绕过前端的接口验证用例；公共 Service 被 Diff 修改时，生成关联旧功能回归用例。
2. **待产品确认**：实现暴露出需求未定义或相互冲突的业务口径；不得采用代码口径替产品回答。
3. **待动态验证**：静态资料无法确认缓存清理、异步消费、事务、并发或真实运行结果；例如 Redis 清理未定位时，生成多账号查询、重新进入页面和等待 TTL 的一致性验证线索；Queue 读取当前比例时，生成修改前后收益结算分界用例。

后端权限校验疑似缺失时，生成普通管理员、跨房间管理员和伪造对象 ID 的手工接口验证用例，不直接写成已确认漏洞或 Bug。每项发现都必须能回指正式用例、待确认编号或待动态验证编号。

## 代码分析停止条件

以下证据达到“足以生成可执行用例”的程度后，停止继续深挖并进入正式测试用例生成：

1. 已定位主要用户入口。
2. 已定位实际接口和主要参数。
3. 已定位关键权限和数据归属校验。
4. 已定位核心状态变化。
5. 已定位主要数据保存位置。
6. 已定位与需求直接相关的缓存或异步链路。
7. 已识别主要前后端和多端差异。
8. 已识别受影响旧功能和回归范围。
9. 已识别静态无法确认、需要动态验证的风险。
10. 已具备生成可执行测试步骤和预期的足够证据。

达到停止条件后，不追踪无关框架底层，不分析第三方库内部实现，不扩展到无明确调用关系的模块，不为每个 `if/else` 生成测试点，也不因“全链路”无限扩大上下文。固定原则：**读到足以提高测试用例准确度即可。**

## 灰盒增强测试设计

保留场景法、等价类、边界值、决策表、状态迁移、权限与数据归属、幂等、并发、数据一致性、前后端一致性、隐含分支、测试点提取和风险优先级。

按风险覆盖：

- 前端与后端限制一致性、绕过前端、客户端可控字段兜底；
- Android/iOS/H5/Admin 参数差异，文档与代码差异；
- 对象级权限、数据归属、数据库字段与唯一约束；
- 老数据、缺字段、空数据、脏数据、迁移兼容；
- Redis 更新/失效、Queue 延迟/重试/重复/补偿、WebSocket/HTTP 同步；
- 事务、锁、幂等、并发、公共方法/模型影响和 Diff 回归；
- 页面、接口、数据库、缓存、消息和下游闭环；
- 多账号、多角色、多端交互、失败恢复、成功但状态未变、失败但状态已变和多数据源不一致。

禁止为每个分支生成一条用例、机械堆 null/空值/超长值、为显得全面而堆低价值场景，或把所有代码疑点写成正式问题。

优先级沿用需求工作台标准并逐条按风险分级：P0 仅用于主流程不可用、越权/数据泄露、资损、数据污染、核心状态错误或会阻断业务的场景；P1 用于重要异常、边界、状态流转、数据一致性和常见失败；P2 用于低频边界、展示/兼容性和低风险补充。P0 不得因属于本轮核心功能而批量泛化；同一风险的等价场景不重复占用 P0。输出前统计 P0/P1/P2，若 P0 明显占多数，必须逐条复核并降回实际风险等级。

需求未定义时间筛选的具体开闭区间时，优先验证“列表与导出时间边界一致”，具体开闭区间待确认；不得直接把 SQL 比较符当作产品时间边界。需求未定义无数据交互时，无数据时只保留一个可判定预期，写明口径待确认和必须满足的不泄露/不产生错误任务等风险目标，不得并列两个互斥结果。

## 历史用例、历史问题与待动态验证

历史资料只在与当前需求直接相关时参与。相关历史 Bug 进入历史回归候选，相关旧用例用于去重和补漏；同接口、同代码位置、同根因不得重复生成。已失效逻辑不进入正式用例，无关历史问题直接排除，历史问题不能替代当前需求规则。

静态资料不能确认的 Redis 刷新、Queue 重复消费、WebSocket 漏推、并发冲突、事务回滚、环境版本、真实请求或底层状态，标记为“待动态验证”。每条必须包含静态证据、风险、执行方式、关联用例、所需账号/数据和观察结果；不得直接标记为Bug。

固定规则：

1. 用例 ID 使用阿拉伯数字，从 1 开始连续；模块分割行不占编号。
2. 允许模块分割行。第一列写“【模块分割行】”，第二列写“第 N 模块：模块名称”，第三列写覆盖范围，其余业务字段写“-”，备注写“模块起始分割”。
3. 验证功能点用 1、2、3、4 编号，写清规则、异常、边界、权限、状态或一致性目标；不得写“功能正常”等空话，也不得复制步骤。
4. 前置条件必须可准备，写明角色、账号、数据、状态、配置和入口；每条用例从独立初始态开始，禁止依赖上一条用例的终态。
5. 测试步骤按手工执行顺序编号，写清入口、动作、关键输入和重查方式；不得把类名、方法名或代码行号作为普通测试人员的操作步骤。
6. 预期结果逐步对应可观察结果，并包含可判定的最终业务结果；不得只写“正常”“成功”“失败”或“符合预期”。
7. 优先级只允许 P0、P1、P2，分别对应核心/高风险、重要异常与边界、低风险补充。
8. 新生成用例的执行结果默认填写“未执行”。状态合同为“未执行 / 通过 / 不通过 / 待定”；后三种仅用于后续执行回填，失败原因、证据路径和实际执行现象属于执行报告或证据回填 Skill。
9. 备注用于需求来源、实现证据、合理假设、待确认、历史回归、待动态验证和本轮不纳入；代码位置只能放在备注或证据摘要，不应成为手工步骤主体。
10. 不新增需求 ID、关联测试点、来源处理、自动化优先级、实际结果等正式列，也不同时输出 8 列、10 列、11 列多个版本。
11. 候选用例按风险去冗余：合并同路径同根因的低价值重复场景，保留不同业务路径、权限边界、关键状态迁移、数据污染风险和可诊断失败。
12. Excel 全表使用黑体 `SimHei`，冻结表头，保留单元格换行、模块分割行、优先级样式和执行状态下拉；HTML 使用相同字段和数据，支持搜索、筛选、冻结表头、状态统计、状态回填及按报告隔离的 `localStorage`，且不得请求外部资源。
13. 文件生成后必须校验：主 Sheet 名、精确表头、列顺序、行列长度、连续 ID、默认状态、字体、可编辑性、Excel 可打开、HTML 可离线浏览和两种格式内容一致。

旧格式只作为输入兼容：

- 旧 8 列输入按“用例 ID、所属模块、用例标题、前置条件、测试步骤、预期结果、优先级、执行结果”识别；转换时补充编号化“验证功能点”和“备注”，重新生成从 1 开始的连续 ID，并把执行结果重置为“未执行”。
- 旧 11 列输入按旧 10 列加“实际结果”识别；转换时移除“实际结果”，统一执行结果表头，重新生成连续 ID，并把新正式用例执行结果重置为“未执行”。
- 旧普通 10 列中“执行结果”表头可被识别；重新生成时统一改为“执行结果（通过 / 不通过 / 未执行）”。

### 手工执行导向

每条正式用例必须明确可准备的角色、账号、测试数据和状态，写清入口与操作路径，并让每一步对应可观察结果和最终业务核验。涉及异步时写明合理等待和重查方式；涉及多账号时写明角色和切换方式；涉及接口绕过时写明请求目标与关键参数，但不要求测试人员阅读源代码。

只有用户具备数据库、Redis、MQ 等访问条件时，才把底层检查写成正式核验步骤；否则改用页面、接口或 Admin 可观察证据，或列入待动态验证。类名、方法名、代码行号只能放入备注或证据摘要，不得作为测试步骤主体。

## 输出分层与文件交付

默认输出顺序固定为：

1. 输入质量与准入结论。
2. 当前需求基线摘要。
3. 高价值需求与实现差异摘要。
4. 正式测试用例 Excel。
5. 聊天中的核心用例说明。
6. 回归范围。
7. 待确认问题。
8. 待动态验证线索。
9. 未覆盖风险。

正式测试用例是第一核心交付物。默认不先输出完整代码调用链、完整需求—实现矩阵、所有类和方法清单、大量代码风险说明或大量独立分析文件。

只有用户明确要求“完整分析”“全量灰盒分析”“完整需求—实现对照”“完整影响链”或“完整代码证据”时，才展开完整对照矩阵、影响链、多端差异、数据/异步链路、追踪矩阵和待动态验证清单；即使进入完整模式，最终仍只生成一套正式用例。

凡生成正式用例，必须以同一份报告 JSON 生成 `.xlsx` 和 `.html`：

- 默认一个 Excel；主 Sheet 是“正式测试用例”。高价值辅助 Sheet 只使用“需求与实现差异”“回归范围”“待动态验证”“证据覆盖”中的必要项，不创建空 Sheet。
- 报告 JSON 设置 `excel_font: "SimHei"`，Excel 字体默认黑体（SimHei），必须可编辑。
- 生成报告前读取 `assets/graybox-report.schema.json` 和 `assets/graybox-report-template.json`；前者是独立输出 schema，后者是可直接复制修改的最小模板。不得删除或改名这两个自包含资产。
- HTML 是单文件离线总览，支持搜索、模块/优先级/状态筛选、冻结表头、状态下拉、统计和按报告隔离的 `localStorage`，不得请求外部资源。
- 状态合同为 `未执行 / 通过 / 不通过 / 待定`；正式表头仍严格使用“执行结果（通过 / 不通过 / 未执行）”。“待定”只在已执行但证据或口径不足时使用。
- Excel/HTML 都能打开并通过 Sheet、表头、列数、内容完整性和可编辑性校验后才能宣称交付完成。

生成命令：

```bash
node scripts/render-test-assets.mjs --input report.json --output-dir output --basename 灰盒增强测试用例
```

Node Renderer 成功后，必须用可用 Python 运行真实单元格校验，不得只检查 ZIP、样式或 Sheet 名：

```bash
python scripts/render-test-assets.py --verify-only --input report.json --xlsx output/灰盒增强测试用例.xlsx --html output/灰盒增强测试用例.html
```

Node 不可用时，只能使用包内受控备用生成器，不得临时编写或改造 Excel 生成逻辑：

```bash
python scripts/render-test-assets.py --input report.json --output-dir output --basename 灰盒增强测试用例
```

备用生成器必须把 `columns` 作为一整行、每个 `row.values` 作为一整行写入二维矩阵，并在落盘前用真实 Excel 解析器逐 Sheet 比对 JSON。Python 或 `openpyxl` 不可用、任一单元格/尺寸不一致或 HTML 不同源时，明确失败并保留 JSON/已有 HTML；不得生成损坏 Excel，也不得声称 Excel 校验通过。
