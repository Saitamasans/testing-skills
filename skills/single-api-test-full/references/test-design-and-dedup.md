## 十、用例生成分层

### 10.1 用例类型

| 用例类型    | 来源          | 是否进入正式用例表 | 预期写法              |
| ------- | ----------- | --------: | ----------------- |
| 确定型用例   | 文档明确        |         是 | 写明确预期             |
| 契约型用例   | 请求 / 响应结构   |         是 | 写明确结构断言           |
| 风险型用例   | 接口类型 + 测试经验 |         是 | 写风险预期，必要时标待确认     |
| 合理假设型用例 | 字段 / 接口名推断  |       可进入 | 必须备注“合理假设”        |
| 待确认型用例  | 依赖业务口径      |  可进入或单独列出 | 不写死 code/msg/业务结果 |
| 禁止型内容   | 无依据猜测       |       不进入 | 不输出               |

### 10.2 正常用例生成规则

每个可测接口至少生成 1 条 P0 正常用例。

正常用例必须校验：

1. HTTP 状态码。
2. 业务 code。
3. msg。
4. response schema。
5. 核心返回字段。
6. 字段类型。
7. 字段取值范围。
8. 是否返回敏感信息。
9. 若接口有数据副作用，应验证写入、修改、扣减、发放、状态变化、流水等。

### 10.3 异常用例生成规则

异常用例应覆盖：

1. 必填字段缺失。
2. 字段为空。
3. 字段为 null。
4. 字段类型错误。
5. 字段格式错误。
6. 字段边界值。
7. 非法枚举。
8. Header 缺失。
9. 鉴权失败。
10. 权限不足。
11. 越权访问。
12. 重复提交。
13. 并发请求。
14. 服务异常响应结构。

错误码缺失时不得编造具体 code/msg，统一使用“十九、错误码缺失时的固定表达”。

## 十一、用例去冗余规则

### 11.1 基本原则

**普通字段校验可以合并，核心业务风险必须拆开。**

### 11.2 拆分规则

| 场景             |         是否单独拆用例 |
| -------------- | --------------: |
| 正常主流程          |              必须 |
| 鉴权失败           |              必须 |
| token 过期 / 无效  |              必须 |
| 越权访问           |              必须 |
| 金额 / 余额 / 资产字段 |              必须 |
| 状态字段           |              必须 |
| 库存 / 次数 / 限制字段 |              必须 |
| 幂等风险           |              必须 |
| 并发风险           |              必须 |
| 数据副作用          |              必须 |
| 普通文本字段为空       |             可合并 |
| 普通文本字段超长       |             可合并 |
| 普通字段类型错误       |             可合并 |
| 非核心非必填字段异常     |             可合并 |
| 错误码缺失          | 不强行拆错误码用例，标记待确认 |

### 11.3 用例数量控制

| 接口复杂度          | 聊天版展示数量 | 表格文件用例数量建议 |
| -------------- | ------: | ---------: |
| 简单查询接口         |   5～8 条 |     8～15 条 |
| 普通新增 / 修改接口    |  8～12 条 |    15～30 条 |
| 资产 / 金额 / 状态接口 | 10～15 条 |    20～40 条 |
| 支付 / 发奖 / 回调接口 | 12～18 条 |    30～60 条 |
| 资料不足场景         |  展示问题为主 |   只生成可确认部分 |

数量不是硬性堆满，必须以风险和可执行性为准。
