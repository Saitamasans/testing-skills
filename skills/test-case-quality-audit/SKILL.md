# 测试用例质量审计与评审门禁 Skill V1

## 1. Skill 定位

**测试用例质量审计与评审门禁 Skill**，用于在测试用例生成后、人工评审和测试执行前，对 GPT 或人工编写的测试用例进行质量审计。

V1 目标：

> 让用例在进入人工评审和测试执行前，先经过一次可靠、克制、可追踪的质量审计。

本 Skill 的核心不是继续生成更多用例，而是判断现有用例是否可靠、是否可执行、是否存在明显遗漏、冗余、冲突、不可判定、不可追踪等问题。

---

## 2. 使用位置

本 Skill 位于测试工作台流程中的第二道门禁：

| 顺序 | 阶段 | 产物 |
|---|---|---|
| 1 | 需求分析门禁 | 产品核对问题表、需求风险表 |
| 2 | 产品 / 业务 / 研发补充确认 | 已确认规则、待确认规则、范围边界 |
| 3 | 测试点提取 | 功能点、异常点、边界点、权限点、数据一致性点 |
| 4 | 测试用例生成 | 初版测试用例表 |
| 5 | **用例质量审计与评审门禁** | 用例问题清单、漏测风险、冗余合并建议、门禁结论 |
| 6 | 用例修订 | 修订版用例表 |
| 7 | 人工最终评审 | 最终执行版用例 |
| 8 | 测试执行 | 执行结果、缺陷、风险记录 |

## 3. 触发条件

当用户出现以下表达时，启用本 Skill：

| 用户表达 | 执行动作 |
|---|---|
| 评审这批用例 | 启用标准用例评审 |
| 按用例评审 Skill 跑一遍 | 启用完整审计流程 |

| 看看有没有漏测 | 在有需求依据时审漏测；无需求时必须降级说明 |
| 看看有没有冗余 | 审重复、低价值、可合并用例 |
| 对照需求审用例 | 强制做需求一致性和覆盖审计 |
| 审完后给我修订建议 | 输出问题清单和修改建议 |
| 根据评审结果修订用例 | 进入修订模式；否则默认不重写用例 |

## 4. 第一原则：先判断审计依据是否完整

用例评审必须先判断输入质量。

### 证据铁律

> **没有需求地基时，不能强行判断覆盖率、漏测、需求冲突或预期正确性。**

任何覆盖、漏测、冲突、预期对错结论都必须能指向需求原文、已确认产品口径、接口契约、状态/权限规则、变更记录或需求—测试点—用例追踪矩阵。找不到依据时，结论只能是“不可判定”或“待确认”，不得把常见经验、实现现状或审计者推测冒充需求。

| 输入情况 | 审计级别 | 允许审计内容 | 禁止事项 |
|---|---|---|---|
| 有需求 + 产品确认 + 用例 | 完整审计 | 需求一致性、覆盖完整性、漏测、冲突、边界、权限、数据一致性、冗余、可执行性 | 不允许脑补需求外规则 |
| 有需求 + 用例 | 标准审计 | 对照需求审覆盖、冲突、异常、边界、方法使用、可执行性 | 产品未确认的规则必须标为“待确认” |
| 只有用例 | 降级审计 | 格式、前置条件、步骤、测试数据、预期清晰度、可执行性、冗余、优先级合理性 | 禁止判断完整覆盖需求；禁止硬判漏测；禁止凭空判定预期正确或错误 |
| 只有需求，没有用例 | 不进入用例评审 | 提醒应先生成用例或提供用例表 | 不做用例审计 |

强制规则：

> 当审计依据不足时，优先暴露信息缺口，而不是强行输出确定性判断。

只有用例、没有需求时，可以指出用例自身的表达或执行问题，也可以给出带推导来源的“待确认候选测试方向”；不得把候选方向改名为“已确认漏测”，不得计算没有分母的覆盖率，不得硬判现有业务预期正确或错误。

如果输入不完整，必须在输出开头明确说明：

| 审计风险声明 | 说明哪些判断依赖缺失信息，不能作为最终执行依据 |
| 信息缺口表 | 列出缺失的需求、规则、接口、权限、状态、数据口径 |

## 6. 核心审计维度

| 维度 | 审计重点 |
|---|---|
| 需求一致性 | 用例是否和需求原文、产品确认口径、需求变更说明冲突 |
| 覆盖完整性 | 是否覆盖主流程、异常流程、边界、权限、状态流转、数据一致性 |
| 场景方法使用 | 是否正确使用场景法、等价类、边界值、决策表、状态迁移 |
| 正反向覆盖 | 是否只有成功路径，缺少失败、拦截、异常提示、中断流程 |
| 边界值覆盖 | 金额、时间、次数、库存、长度、枚举、状态边界是否覆盖 |
| 权限与越权 | 未登录、无权限、越权、跨用户、角色差异是否覆盖 |
| 前后端一致性 | 前端限制和后端校验是否都被考虑，是否存在绕过前端直接调接口的风险 |
| 数据一致性 | 页面展示、接口返回、数据库状态、余额、库存、订单、记录是否闭环 |
| 并发与幂等 | 连点、重复提交、重复回调、重复领取、并发扣减是否覆盖 |
| 可执行性 | 前置条件、测试数据、步骤、账号、环境、初始状态是否明确 |
| 可判定性 | 预期结果是否能判断通过 / 失败，不能只写“正常”“成功”“符合预期” |
| 冗余与合并 | 是否存在重复、低价值、可合并用例；相似不等于重复，合并需保守 |
| 优先级合理性 | P0/P1/P2 是否与核心链路、业务风险、影响范围匹配 |
| 需求缺口识别 | 审计过程中发现哪些信息不足以支撑用例判断 |

## 7. 审计流程

| 步骤 | 动作 | 输出 |
|---|---|---|
| 1 | 检查输入完整性 | 审计级别、降级说明、信息缺口 |
| 2 | 锁定需求版本和产品口径 | 当前审计依据、待确认规则 |
| 3 | 检查用例结构 | 字段完整性、ID、模块、优先级、步骤、预期 |
| 4 | 对照需求审覆盖 | 覆盖项、疑似漏测、需求冲突 |
| 5 | 审方法使用 | 等价类、边界值、场景法、决策表、状态迁移 |
| 6 | 审用例质量 | 可执行性、可判定性、测试数据、前置条件 |
| 7 | 审风险场景 | 权限、越权、并发、幂等、数据一致性、前后端一致性 |
| 8 | 审冗余重复 | 重复用例、低价值用例、保留与合并建议 |
| 9 | 审需求变更影响 | 新增、删除、修改、废弃用例风险 |
| 10 | 输出门禁结论 | 可进入人工评审、修订后进入、不建议执行、退回需求确认 |

### 7.1 可跟踪审计清单

每次审计按下表逐项记录。聊天交付可使用精简表述，但不得省略“状态、证据、结论/问题 ID”的追踪关系；不适用项标记 `N/A` 并说明原因。

- [ ] 检查输入完整性并记录信息缺口 ID ⚠️ REQUIRED
- [ ] 判定唯一审计级别并说明原因 ⚠️ REQUIRED
- [ ] 锁定需求、用例和产品口径版本
- [ ] 检查用例结构并关联用例 ID、问题 ID
- [ ] 依据需求或测试点证据执行覆盖审计
- [ ] 检查测试方法使用及适用依据
- [ ] 检查风险场景并记录影响、证据和阻塞状态
- [ ] 保守检查冗余，证据不足时不贸然删并
- [ ] 输出可回溯到问题 ID 和证据的门禁结论 ⚠️ REQUIRED

| 检查项 | 必查内容 | 记录要求 |
|---|---|---|
| 输入完整性 | 需求、产品确认、接口/规则、用例是否齐全 | 记录已提供项、缺失项及信息缺口 ID |
| 审计级别 | 完整审计、标准审计、降级审计、不进入用例评审 | 明确唯一等级及判定原因 |
| 版本口径 | 需求版本、用例版本、产品补充口径、变更记录 | 记录版本；未知时标“待确认”，不得默认最新 |
| 用例结构 | ID、模块、优先级、前置、数据、步骤、预期 | 每个问题关联用例 ID 和问题 ID |
| 覆盖审计 | 主流程、异常、边界、权限、状态、数据一致性 | 关联具体需求/测试点证据；无证据时标不可判定 |
| 方法使用 | 场景法、等价类、边界值、决策表、状态迁移 | 说明适用方法、使用情况和依据 |
| 风险审计 | 资损、权限、并发、幂等、前后端、数据一致性 | 记录风险、影响、证据和是否阻塞 |
| 冗余审计 | 重复、低价值、合并损失 | 关联涉及用例 ID；证据不足时保留，不贸然删并 |
| 门禁结论 | 准入结论、阻塞项、待确认项、下一步 | 结论须能回溯到问题 ID 和证据 |

### 7.2 审计与修订边界

> **⛔ BLOCKING：用户未明确要求修订时，不得重写整批用例。默认交付是审计结论、问题清单和修订建议。**

- “评审、审计、检查、评价、看看问题、给修订建议”都不等于授权重写。
- 只有用户明确要求“根据审计结果修订/重写用例”时，才可进入修订模式；必须先完成审计，再按问题 ID 修订，并说明原用例 ID 与修订项的对应关系。
- 即使用户明确要求修订，仍不得编造未确认规则。证据不足的字段保留为“待确认”或带方括号的占位项，并明确不可直接执行。
- 必须区分“审计问题严重级别”和“正式测试用例优先级”：审计问题可依据影响与风险证据判为 P0/P1/P2；若需求、产品口径或测试策略没有提供用例优先级依据，修订后正式用例的优先级字段必须填写“待确认”。P0/P1 只能作为带依据的审计建议单独记录，不能写入正式用例优先级字段。
- 修订范围仅限用户授权的用例；不得顺手扩写整批新用例。候选漏测方向只有在需求依据确认后，才能转成正式补充用例。

## 8. 问题严重级别

问题分级要少而准，不能把低价值格式问题刷成高风险。

| 级别 | 定义 | 示例 |
|---|---|---|
| P0 | 会导致重大误测、漏测、资损、越权、核心流程缺失，阻塞进入执行 | 支付扣款无校验；跨用户越权未覆盖；核心状态流转缺失；用例预期与需求冲突 |
| P1 | 影响重要场景覆盖或用例执行质量，需要修订后再评审 | 边界值缺失；异常流程不完整；预期不可判定；测试数据缺失 |
| P2 | 不阻塞执行，但影响评审效率、维护性或表达质量 | 标题不规范；备注缺失；步骤表达可优化；部分重复可合并 |
| 建议 | 不影响执行，仅作为质量增强 | 自动化候选标记、历史风险补充、优先级微调 |

## 9. 门禁结论规则

| 结论 | 判定标准 |
|---|---|
| 可进入人工评审 | 无 P0 问题，P1 问题少且不影响核心链路判断 |
| 修订后进入人工评审 | 无重大需求冲突，但存在 P1/P2 问题，需要修订 |
| 不建议进入执行 | 存在核心流程缺失、需求冲突、资损/权限/数据一致性高风险漏测、大量不可执行或不可判定用例 |
| 需退回需求确认 | 关键规则缺失，导致无法判断用例预期或覆盖是否正确 |
| 仅完成降级审计 | 只有用例缺少需求依据，只能审用例自身质量，不能给完整执行准入结论 |

阻塞项示例：

| 阻塞类型 | 示例 |
|---|---|
| 核心需求无对应测试用例 | 支付成功、扣款、发货、退款状态均无覆盖 |
| 预期结果与需求冲突 | 需求说不可重复领取，用例预期写允许重复领取 |
| 高风险场景缺失 | 资损、越权、并发、幂等、数据一致性未覆盖 |
| 大量用例不可执行 | 无账号、无数据、无环境、步骤不清 |
| 大量预期不可判定 | 只写“成功”“正常”“符合预期” |
| 关键规则待确认但用例写死 | 产品未确认金额上限，用例直接写死最大值 |

## 10. 冗余判断原则

冗余判断必须保守。
相似用例不一定是重复用例。

| 不建议合并的情况 | 原因 |
|---|---|
| 不同业务路径 | 主流程、异常流程、中断流程不能强行合并 |
| 不同风险等级 | P0 和 P2 不能为了减少数量强行合并 |
| 不同权限角色 | 管理员、普通用户、游客、无权限用户不能随意合并 |
| 不同状态迁移 | 待支付、已支付、已取消、已退款等状态应分开判断 |
| 不同错误码或提示 | 接口诊断价值不同，不宜合并 |
| 数据可能互相污染 | 金额、库存、余额、订单、奖励类用例需谨慎合并 |

冗余建议必须说明：

1. 涉及哪些用例；
2. 为什么认为冗余；
3. 建议保留哪条；
4. 合并后是否会损失风险覆盖。

---

## 11. 漏测判断原则

漏测建议必须带依据，不能凭空喊“缺少并发”“缺少边界”。

| 要求 | 说明 |
|---|---|
| 必须说明推导来源 | 来自需求明示、业务动作、状态流转、接口契约、权限规则、历史风险 |
| 必须说明风险 | 为什么这个漏测会影响质量或上线 |
| 必须给出建议补充方向 | 建议补什么类型的用例，而不是泛泛而谈 |
| 缺少需求依据时必须降级 | 只有用例时，不能完整判断漏测，只能提示“可能需要结合需求确认” |

## 12. 默认禁止事项

| 禁止事项 | 说明 |
|---|---|
| 禁止在无需求依据时硬判漏测 | 没有需求地基，不能判断完整覆盖 |
| 禁止凭空改写产品规则 | 未明确规则只能标待确认 |

| 禁止为了显得专业堆低价值问题 | 问题要少而准，优先 P0/P1 |
| 禁止把“实现现状”直接当需求 | 研发实现和产品需求不一致时，应标记待确认 |
| 禁止把相似用例简单判为重复 | 需判断业务路径、状态、权限、风险是否一致 |
| 禁止输出没有依据的覆盖率结论 | 覆盖率必须基于需求、测试点或追踪矩阵 |

### 14.1 聊天文字版结构

聊天中优先输出精简结论，不要堆砌全部明细。

| 顺序 | 内容 |

| 1 | 审计依据完整性结论 |
| 2 | 本次能审什么 / 不能审什么 |

### 14.2 表格文件版结构

只有触发文件交付条件时才进入本节。表格文件应适合评审流转，不是简单复制聊天内容。
建议输出为 Excel / WPS 可打开的表格文件，包含以下 8 个 Sheet；用户另有明确模板时按用户模板生成：

| Sheet | 内容 |
|---|---|
| 01_评审总览 | 审计级别、输入完整性、需求版本、用例总数、问题数量、门禁结论 |
| 02_用例问题清单 | 用例 ID、问题类型、严重级别、问题描述、影响、修改建议、责任对象、处理状态、是否阻塞 |
| 03_漏测风险清单 | 漏测 ID、所属模块、漏测类型、推导依据、风险描述、风险级别、建议补充用例 |
| 04_冗余合并建议 | 冗余组 ID、涉及用例 ID、冗余原因、处理建议、建议保留用例、合并风险 |
| 05_待确认问题 | 问题 ID、确认对象、关联用例、问题描述、需要确认的问题、影响范围 |
| 06_方法使用审计 | 场景法、等价类、边界值、决策表、状态迁移、权限矩阵使用情况 |
| 07_需求变更影响 | 变更类型、影响用例、处理建议、是否阻塞 |
| 08_门禁结论 | 最终结论、阻塞原因、修订优先级、是否建议进入人工评审 |

## 17. 最终自检清单

输出前必须自检：

| 自检项 | 要求 |
|---|---|
| 是否先判断输入完整性 | 必须 |
| 是否明确审计级别 | 必须 |
| 是否说明本次不能审什么 | 输入不完整时必须 |
| 是否避免无依据脑补 | 必须 |
| 是否优先输出 P0/P1 | 必须 |
| 是否区分审计问题严重级别与正式测试用例优先级 | 必须 |
| 缺少产品/测试策略优先级口径时，正式用例优先级是否填写“待确认” | 必须；P0/P1 只能作为带依据的建议，不能写入正式字段 |
| 漏测建议是否带依据 | 必须 |
| 冗余判断是否保守 | 必须 |
| 是否默认不重写用例 | 必须 |
| 是否给出门禁结论 | 必须 |
| 未明确要求修订时是否只审计、不重写整批用例 | ⛔ BLOCKING |
| 是否默认提供聊天文字版 | 必须 |
| 是否仅在用户明确要求 Excel / WPS、归档或评审流转文件时生成表格文件 | 必须 |
| 文件交付是否适合人工评审流转 | 生成文件时必须 |

## 18. 最终输出口径

每次使用本 Skill 时，最终必须给出明确结论：

| 结论类型 |
|---|
| 可进入人工评审 |
| 修订后进入人工评审 |
| 不建议进入执行 |
| 需退回需求确认 |
| 仅完成降级审计，不能下完整执行结论 |

结论必须基于证据，不得为了显得完整而编造需求、规则、预期或漏测项。

## 19. 互斥路由与组合调用

输入重点是已有测试用例，且用户要求审计、评审、查漏、去冗余或修订建议时，本 Skill 作为主 Skill；需求尚未形成用例时改走需求澄清或需求工作台。

审计对象是正式服执行用例时，可建议正式服验证作为最多一个辅助 Skill。调用前先展示主 Skill、辅助 Skill、职责分工和“最终只生成一套审计结果”，暂停等待用户确认。用户确认前，不加载辅助 Skill、不执行生产安全专项分析；拒绝后继续本 Skill，并明确本次结论不包含正式服执行安全门禁审查。聊天或审计总览记录调用与确认结果。

本 Skill 不生成正式测试用例，因此不套用用例执行状态下拉，也不因全局双交付规则自动生成 `.xlsx` 和 `.html`；只有用户单独指定审计报告载体时，才按指定格式交付审计结果。
